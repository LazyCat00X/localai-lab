1. What Is Open WebUI?

Open WebUI is the most popular self-hosted AI interface platform, with 143,000+ GitHub stars and over 16,800 commits. It provides a ChatGPT-like web interface that connects to any OpenAI-compatible backend — Ollama, vLLM, LiteLLM, OpenAI, Groq, Mistral, OpenRouter, and more.

Beyond basic chat, Open WebUI supports:

  • Local RAG — Upload documents and chat with them using your choice of 9 vector databases
  • Voice/Video calls — Hands-free interaction with multiple STT/TTS providers
  • Image generation — Built-in support for DALL-E, ComfyUI, and AUTOMATIC1111
  • Model Builder — Create and customize Ollama models directly from the web UI
  • RBAC — Role-based access control with LDAP/SCIM support for teams
  • Web search — 15+ search providers for RAG-enhanced web queries
  • Pipelines plugin system — Custom Python logic, rate limiting, monitoring, and more
Architecture: Open WebUI is built with SvelteKit (frontend) and Python FastAPI (backend). It runs as a single Docker container with optional database backends — SQLite by default, or PostgreSQL for production deployments.

2. Docker Installation

Quick Start (Ollama on the same host)

If Ollama is running on the same machine, the simplest deployment uses --add-host to allow the container to reach the host's Ollama server:

$ docker run -d -p 3000:8080 \
    --add-host=host.docker.internal:host-gateway \
    -v open-webui:/app/backend/data \
    --name open-webui \
    --restart always \
    ghcr.io/open-webui/open-webui:main

Access the web interface at http://localhost:3000. The first account you create becomes the admin.

With Bundled Ollama (Single Container)

Open WebUI publishes a combined image with Ollama baked in:

# With GPU support
$ docker run -d -p 3000:8080 \
    --gpus=all \
    -v ollama:/root/.ollama \
    -v open-webui:/app/backend/data \
    --name open-webui \
    --restart always \
    ghcr.io/open-webui/open-webui:ollama

# CPU only
$ docker run -d -p 3000:8080 \
    -v ollama:/root/.ollama \
    -v open-webui:/app/backend/data \
    --name open-webui \
    --restart always \
    ghcr.io/open-webui/open-webui:ollama

With NVIDIA GPU Acceleration

$ docker run -d -p 3000:8080 \
    --gpus all \
    --add-host=host.docker.internal:host-gateway \
    -v open-webui:/app/backend/data \
    --name open-webui \
    --restart always \
    ghcr.io/open-webui/open-webui:cuda

For CUDA, ensure the NVIDIA CUDA container toolkit is installed on your Linux host.

For OpenAI API Only (No Ollama)

$ docker run -d -p 3000:8080 \
    -e OPENAI_API_KEY=sk-your-key-here \
    -v open-webui:/app/backend/data \
    --name open-webui \
    --restart always \
    ghcr.io/open-webui/open-webui:main

3. Connecting AI Backends

Open WebUI can connect to multiple backends simultaneously. Configure them in the admin panel under Admin Settings → Connections.

Ollama (Local)

By default, Open WebUI connects to http://host.docker.internal:11434. If Ollama is on a different server:

$ docker run -d -p 3000:8080 \
    -e OLLAMA_BASE_URL="https://ollama.example.com" \
    -v open-webui:/app/backend/data \
    --name open-webui \
    --restart always \
    ghcr.io/open-webui/open-webui:main

OpenAI / LiteLLM / Other Providers

In the admin panel, add an OpenAI-compatible endpoint:

Multiple Model Conversations

Open WebUI supports chatting with multiple models simultaneously. Select multiple models from the dropdown — each model receives the same messages and displays its response in parallel. This is useful for comparing outputs across different providers or model sizes.

4. pip Installation (No Docker)

If you prefer not to use Docker, install Open WebUI directly with Python 3.11:

$ pip install open-webui
$ open-webui serve

This starts the server on http://localhost:8080. All data is stored in ~/.open-webui/ by default.

Python version: Open WebUI requires Python 3.11 for compatibility. Using Python 3.12 or 3.13 may cause dependency conflicts with the bundled vector database libraries.

9. Docker Compose Deployment

For a production setup with PostgreSQL and Ollama:

# docker-compose.yml
services:
  ollama:
    image: ollama/ollama:latest
    volumes:
      - ollama_data:/root/.ollama
    ports:
      - "11434:11434"
    deploy:
      resources:
        reservations:
          devices:
            - driver: nvidia
              count: all
              capabilities: [gpu]

  open-webui:
    image: ghcr.io/open-webui/open-webui:main
    ports:
      - "3000:8080"
    volumes:
      - open-webui_data:/app/backend/data
    environment:
      - OLLAMA_BASE_URL=http://ollama:11434
      - WEBUI_SECRET_KEY=${WEBUI_SECRET_KEY:-change-me-in-production}
      - ENABLE_SIGNUP=false
    depends_on:
      - ollama
    restart: unless-stopped

volumes:
  ollama_data:
  open-webui_data:

Deploy with:

$ docker compose up -d
$ docker compose logs -f open-webui
Production Note: Always set WEBUI_SECRET_KEY to a strong random value in production. Generate one with openssl rand -hex 32. Disable signup (ENABLE_SIGNUP=false) and create users manually or via SCIM/LDAP.

10. Pipelines Plugin Framework

Open WebUI's Pipelines framework lets you inject custom Python logic into the chat pipeline. This is how you add rate limiting, content filtering, usage monitoring with Langfuse, real-time language translation, and toxic message filtering — all without modifying Open WebUI's core code.

Setting Up Pipelines

Pipelines run as a separate service that Open WebUI connects to as an OpenAI-compatible endpoint:

$ docker run -d -p 9099:9099 \
    -v pipelines:/app/pipelines \
    --name pipelines \
    --restart always \
    ghcr.io/open-webui/pipelines:main

Then configure Open WebUI to use the Pipelines URL as an additional OpenAI endpoint. In the admin panel, add http://pipelines:9099 (or http://localhost:9099 for host-network setups) as a new OpenAI connection.

Example: Custom Rate Limiting Filter

# pipelines/rate_limit.py
import time
from typing import Dict

class Pipeline:
    def __init__(self):
        self.rate_limit = 10  # requests per minute
        self.tokens: Dict[str, list] = {}

    async def on_startup(self):
        print(f"Rate limit pipeline started: {self.rate_limit}/min")

    async def on_chat(self, body: dict, user: dict) -> dict:
        uid = user.get("id", "anonymous")
        now = time.time()
        window = now - 60
        if uid not in self.tokens:
            self.tokens[uid] = []
        self.tokens[uid] = [t for t in self.tokens[uid] if t > window]
        if len(self.tokens[uid]) >= self.rate_limit:
            return {"error": "Rate limit exceeded. Try again later."}
        self.tokens[uid].append(now)
        return body

Image Generation Integration

Open WebUI supports image generation directly in chat using multiple backends. Configure in Admin Settings → Image Generation:

BackendSetupCapabilities
OpenAI DALL-ESet OPENAI_API_KEYText-to-image, prompt-based editing
ComfyUI (local)Point to http://comfyui:8188Full workflow-based generation with custom nodes
AUTOMATIC1111 (local)Point to http://localhost:7860Stable Diffusion WebUI integration

For local ComfyUI integration, ensure ComfyUI is running with --enable-cors-header and accessible from the Open WebUI container:

$ docker run -d --gpus all -p 8188:8188 \
    -v comfyui_data:/workspace \
    --name comfyui \
    --restart always \
    comfy:latest

Then in Open WebUI, set the ComfyUI API URL to http://comfyui:8188 (with Docker networking) or http://host.docker.internal:8188.

Observability with OpenTelemetry

For production deployments, Open WebUI supports OpenTelemetry for traces, metrics, and logs. Enable it by setting environment variables:

$ docker run -d -p 3000:8080 \
    -e OTEL_SDK_DISABLED=false \
    -e OTEL_EXPORTER_OTLP_ENDPOINT=http://otel-collector:4318 \
    -e OTEL_SERVICE_NAME=open-webui \
    -v open-webui:/app/backend/data \
    --name open-webui \
    --restart always \
    ghcr.io/open-webui/open-webui:main

Pair this with a collector and an observability backend like Grafana Tempo or Jaeger for full request tracing. The metrics capture generation latency, token throughput, error rates, and active user sessions.

Network Mode Alternative

If host.docker.internal doesn't work on your system (common on Linux Docker without Docker Desktop), use --network=host:

$ docker run -d --network=host \
    -v open-webui:/app/backend/data \
    -e OLLAMA_BASE_URL=http://127.0.0.1:11434 \
    --name open-webui \
    --restart always \
    ghcr.io/open-webui/open-webui:main

Note: With --network=host, the port changes from 3000 to 8080.

Updating Open WebUI

$ docker pull ghcr.io/open-webui/open-webui:main
$ docker stop open-webui
$ docker rm open-webui
# Re-run the docker run command from above
