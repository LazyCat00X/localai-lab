1. What Is Quantization?

Quantization is the single most important technique for running large language models on consumer hardware. Without it, even a modest 7B-parameter model would demand ~14 GB of VRAM in FP16 — well beyond most people's GPUs. With quantization, that same model runs comfortably on a 6 GB GTX 1060.

At its core, quantization reduces the precision of a model's weights — the millions (or billions) of floating-point numbers that encode what the model has learned. FP16 weights use 16 bits each; a Q4_K_M quant uses roughly 4.5 bits each. That's a ~3.6× reduction in memory footprint, and because memory bandwidth is usually the bottleneck for LLM inference, it also means 2–4× higher throughput.

Quantization isn't compression in the zip-file sense. You're not encoding redundant patterns — you're discarding precision. The art of good quantization is deciding which bits to keep and how to arrange them so the model still produces coherent, accurate output. Modern quantization schemes like K-quants and IQ-quants do this with surgical precision, achieving less than 1% perplexity degradation at 4-bit precision compared to FP16.

Key insight: LLM weights are surprisingly robust to noise. A model quantized to Q4_K_M often produces text indistinguishable from FP16 in casual use. The degradation primarily shows up in edge cases — nuanced reasoning, rare knowledge, or tasks requiring exact numerical recall. For chatbots, summarization, and most practical applications, 4-bit is the sweet spot.

2. GGUF Quantization Levels: Complete Reference

The GGUF format (introduced by llama.cpp and now used by Ollama, LM Studio, GPT4All, and dozens of other tools) supports over 30 distinct quantization types. They fall into two families: K-quants (the original, mature scheme) and IQ-quants (newer, importance-weighted variants that achieve better quality-per-bit).

Every quant type is identified by a name like Q4_K_M. Breaking this down:

  • Q4 — The target bit-width (4-bit in this case)
  • _K — Uses the K-quant block-wise scheme
  • _M — Medium variant (vs _S for small and _L for large)

The _S/_M/_L suffixes control how many layers use the higher-precision "super-block" quantization. _S applies it only to the output tensor; _M extends it to the attention layers; _L covers nearly all layers. More super-blocks = better quality at slightly higher bitrate.

When to prefer IQ over K: If you're picking a quant from a reputable publisher who used an imatrix (most HuggingFace GGUF uploads from TheBloke, Bartowski, or MaziyarPanahi include IQ variants), IQ-quants almost always win at the same bitrate. The quality gains are most dramatic at 2–3 bits and diminish above Q5. At Q6_K quality levels, the difference is negligible.

3. K-Quants vs IQ-Quants: How They Work

K-Quants: Block-Wise Quantization

K-quants divide each weight tensor into blocks of 256 weights. Within each block, they find the minimum and maximum value, compute a scale factor, and map all 256 values into the target bit-width range. This works well because nearby weights in a tensor tend to have similar magnitudes — the block's dynamic range is much smaller than the tensor's global range, so per-block scaling captures detail that global scaling would lose.

The critical insight behind K-quants is super-blocks. Instead of quantizing every block uniformly, a super-block keeps certain blocks at higher precision. In Q4_K_M, the attention layers' output projections get super-block treatment with 6-bit quantization for weights that the Hessian analysis identifies as outliers. This costs ~0.5 extra bits on average but recovers most of the quality gap to FP16.

IQ-Quants: Importance-Weighted Grouping

IQ-quants take a fundamentally different approach. Instead of fixed 256-weight blocks, they group weights by importance as measured by a calibration pass. The calibration runs a representative dataset through the FP16 model and computes a Hessian approximation — essentially, "how much does the loss change when I perturb this weight?"

Weights with high importance (the model is very sensitive to their exact values) get allocated to higher-bit groups. Weights with low importance (the loss barely budges when they change) get squeezed into ultra-low-bit groups. The result is a non-uniform bit allocation that achieves significantly better quality-per-bit than any uniform scheme.

The tradeoff: IQ-quants require an imatrix calibration pass, which takes extra time during the quantization process (typically 10–30 minutes for a 7B model). K-quants can be computed without one. For model publishers, this is an upfront cost paid once. For end users downloading GGUF files, it's free quality.

What these numbers mean in practice: A +1% perplexity increase on WikiText-2 is barely noticeable in chatbot use. A +4% increase (Q3_K_M territory) — the model starts making subtle mistakes: confusing similar facts, slightly garbled formatting, occasional nonsensical completions. At +7% (Q2_K), the model is noticeably degraded for anything beyond simple Q&A. For creative writing or code generation, stay at Q4_K_M or above.

It's important to note that perplexity is a proxy metric — it measures how well the model predicts held-out text, which correlates with but doesn't perfectly capture real-world task performance. A model with +2% perplexity might still ace your specific use case. The only definitive test is running your actual workload.

5. Speed vs Quality: The Real Tradeoff

Quantization doesn't just reduce memory — it speeds up inference. But the relationship isn't linear, and the bottleneck shifts depending on your hardware.

CPU-Only Inference (Pure llama.cpp)

On CPU, inference is entirely memory-bandwidth bound. The CPU cores sit idle waiting for weights to stream in from RAM. A lower-bit quant means fewer bytes to read per token, and throughput scales nearly linearly with bit reduction:

QuantGB/s Read per Token (7B)Tokens/sec (DDR5-4800)Tokens/sec (DDR4-3200)
Q8_08.0~5.5 tok/s~3.7 tok/s
Q4_K_M4.5~10.0 tok/s~6.7 tok/s
Q3_K_M3.6~12.5 tok/s~8.3 tok/s
Q2_K2.6~17.0 tok/s~11.5 tok/s

These numbers assume a modern x86_64 CPU (Ryzen 7 or Core i7) with all cores active. Single-channel memory halves throughput. Apple Silicon with unified memory (M2/M3/M4) achieves 2–3× higher tok/s due to much wider memory buses (100–400 GB/s vs 40–70 GB/s for DDR5).

GPU Inference (CUDA/Metal/Vulkan)

On GPU, the picture changes. GPUs have enormous memory bandwidth (RTX 4090: 1,008 GB/s; RTX 3060: 360 GB/s). For small batch sizes (batch=1, the common case for interactive chat), the bottleneck shifts to compute throughput — the GPU can read weights faster than it can process them. In this regime, lower-bit quants can actually be slower because decompressing low-bit weights into FP16 for compute adds overhead:

# Llama 3.1 8B, RTX 3060 12 GB, llama.cpp CUDA build
$ ./build/bin/llama-bench -m llama-3.1-8b-Q4_K_M.gguf -ngl 99
| model               | size     | backend    | ngl | test       | t/s    |
| ------------------- | -------- | ---------- | --- | ---------- | ------ |
| llama 3.1 8B Q4_K_M | 4.92 GiB | CUDA       |  99 | pp 512     | 229.42 |
| llama 3.1 8B Q4_K_M | 4.92 GiB | CUDA       |  99 | tg 128     |  52.31 |

$ ./build/bin/llama-bench -m llama-3.1-8b-Q8_0.gguf -ngl 99
| model               | size     | backend    | ngl | test       | t/s    |
| ------------------- | -------- | ---------- | --- | ---------- | ------ |
| llama 3.1 8B Q8_0   | 8.54 GiB | CUDA       |  99 | pp 512     | 198.65 |
| llama 3.1 8B Q8_0   | 8.54 GiB | CUDA       |  99 | tg 128     |  49.87 |

Notice: Q8_0 is only 5% slower than Q4_K_M on GPU despite being nearly twice the size. That's because both fit entirely in VRAM, and compute throughput (not memory bandwidth) dominates at batch=1. For larger batch sizes (batch=8+), memory bandwidth begins to matter again, and lower-bit quants pull ahead.

Key takeaway: For interactive chat on GPU, choose the highest quality quant that fits in VRAM. Speed differences are small. For batch processing or server deployments, prioritize throughput — lower-bit quants win.

6. Choosing the Right Quant: Decision Framework

There is no "best" quantization — there is only the best quant for your hardware and workload. Use this decision tree:

Q4_K_M: The Universal Default

If you don't know what to pick, pick Q4_K_M. It's the default in Ollama (ollama pull llama3.1:8b downloads Q4_K_M), and for good reason: it offers ~1% quality loss for a ~3.5× size reduction. On any GPU with 8+ GB VRAM or any system with 16+ GB RAM, it runs comfortably.

When to Go Higher (Q5–Q8)

  • You have a 12+ GB GPU (RTX 3060/4070, RX 6800) and care about quality: Q5_K_M adds only ~600 MB but recovers half the quality gap.
  • Exact factual recall matters — Q&A over a knowledge base, RAG pipelines, structured data extraction: Q6_K or Q8_0 is worth the extra VRAM.
  • You're generating embeddings — embedding quality degrades faster than text quality under quantization. Use Q8_0 or FP16 for embedding models.
  • Production API serving — If customers depend on your output, don't cut corners. Q6_K is the minimum for production LLM APIs; Q8_0 is preferred.

When to Go Lower (Q2–Q3)

  • You have 6 GB or less VRAM — Q3_K_M or IQ3_XS can cram an 8B model into 4 GB, leaving room for context.
  • You're running on a phone — MLC Chat and llama.cpp Android builds need every byte. IQ2_XXS can run a 7B model on a 4 GB phone.
  • You want to run a 70B model on a 24 GB GPU — Q3_K_M fits Llama 3.1 70B in ~28 GB. With 8 layers offloaded to CPU, it fits in 24 GB VRAM + 32 GB RAM.
  • Speed is everything, quality is secondary — CPU batch processing, non-critical summarization.

7. How to Quantize Your Own Models

Converting a HuggingFace model to GGUF with your chosen quantization level is straightforward with llama.cpp's llama-quantize tool. Here's the end-to-end workflow:

Step 1: Convert to FP16 GGUF

# Clone and build llama.cpp (if not already)
git clone https://github.com/ggerganov/llama.cpp
cd llama.cpp
cmake -B build
cmake --build build -j $(nproc)

# Convert a HuggingFace model to FP16 GGUF
python convert_hf_to_gguf.py /path/to/huggingface/model \
  --outfile model-f16.gguf \
  --outtype f16
Model availability: You can download models from HuggingFace Hub with huggingface-cli download meta-llama/Llama-3.1-8B-Instruct --local-dir ./Llama-3.1-8B-Instruct. For gated models (Llama, Mistral Large), you'll need to accept the license agreement on the model card page first and authenticate with huggingface-cli login.

Step 2: Quantize to Your Target Format

# Basic quantization (no imatrix)
./build/bin/llama-quantize model-f16.gguf model-Q4_K_M.gguf Q4_K_M

# IQ quantization with importance matrix
# Step 2a: Generate imatrix from calibration data
./build/bin/llama-imatrix -m model-f16.gguf -f calibration.txt \
  -o imatrix.dat --chunk 512

# Step 2b: Quantize using imatrix
./build/bin/llama-quantize --imatrix imatrix.dat \
  model-f16.gguf model-IQ4_XS.gguf IQ4_XS

The calibration data (calibration.txt) should be a representative sample of text the model will encounter. For general-purpose models, the WikiText-2 training set works well (available from HuggingFace: Salesforce/wikitext). For domain-specific models, use your own documents — the imatrix will learn which weights matter for your data distribution.

# Download and prepare WikiText-2 calibration data
wget https://huggingface.co/datasets/Salesforce/wikitext/resolve/main/wikitext-2-raw-v1.zip
unzip wikitext-2-raw-v1.zip
# Combine training files (the model was trained on Wikipedia already; use validation as calibration)
cat wikitext-2-raw/wiki.valid.raw > calibration.txt

Step 3: Verify Quality

# Perplexity evaluation on a held-out test set
./build/bin/llama-perplexity -m model-Q4_K_M.gguf -f test.txt \
  --ctx-size 2048

# Quick sanity check: generate some text
./build/bin/llama-cli -m model-Q4_K_M.gguf \
  -p "Explain the difference between supervised and unsupervised learning:" \
  -n 200 --temp 0.7

GPTQ and AWQ: GPU-Native Alternatives

GPTQ and AWQ are quantization formats designed specifically for GPU inference. Unlike GGUF (which targets universal hardware), GPTQ/AWQ assume a CUDA GPU and optimize for GPU tensor core throughput:

When to use GPTQ/AWQ over GGUF: If you're deploying a vLLM server on NVIDIA GPUs for high-throughput production serving (batch > 4), AWQ via vLLM's Marlin kernel is the current performance leader. For everything else — local dev, single-user chat, CPU inference, Apple Silicon — GGUF is simpler, faster to set up, and more portable.

Ollama: Quantization Without the Complexity

Ollama abstracts quantization entirely. When you pull a model with ollama pull llama3.1:8b, Ollama downloads the Q4_K_M GGUF file automatically. You can request specific quants using tags:

# Pull specific quantization levels
ollama pull llama3.1:8b          # Q4_K_M (default)
ollama pull llama3.1:8b-instruct-q5_K_M
ollama pull llama3.1:8b-instruct-q8_0

# Check what quant you have
ollama list
# NAME                    ID              SIZE      MODIFIED
# llama3.1:8b             421087e2de5c    4.9 GB    2 days ago

To create a custom Modelfile with a specific quant from a local GGUF file:

# Create Modelfile pointing to your local GGUF
echo 'FROM ./llama-3.1-8b-Q5_K_M.gguf
PARAMETER temperature 0.7
PARAMETER num_ctx 8192' > Modelfile

ollama create my-custom-model -f Modelfile
ollama run my-custom-model

9. Practical Tips and Common Pitfalls

Tip 1: Q4_K_M Is Good Enough for Almost Everything

Engineers tend to over-optimize for quality. In blind tests, most people cannot distinguish Q4_K_M from FP16 in chatbot conversations. Unless you're doing highly factual work (medical, legal, math proofs), don't burn VRAM on Q5+ that could be used for longer context windows instead.

Tip 2: Watch Out for "Default" Quants on HuggingFace

Many GGUF uploads tag Q4_K_M as the "recommended" or default download. But check the file size — some publishers (especially for large 70B+ models) default to Q3_K_M or lower to reduce bandwidth. Always verify what you're downloading.

Tip 3: Context Length Eats More VRAM Than You Think

KV cache memory scales with context_length × kv_heads × head_dim × layers × 2 × 2 bytes. For a 7B model at 32K context, that's roughly 2–3 GB of additional VRAM. A model that "fits" with Q4_K_M at 2K context may OOM at 32K. If you need long contexts, drop down one quant level — Q4_K_S + 32K context beats Q5_K_M + 4K context for most applications.

Tip 4: Embedding Models Need Higher Precision

Text embedding models (BGE, E5, Jina) are far more sensitive to quantization than generative LLMs. An embedding model quantized to Q4_K_M can lose 10–20% retrieval accuracy on standard benchmarks. Always use Q8_0 or FP16 for embedding models in RAG pipelines.

Tip 5: Flash Attention and Quantization Don't Always Mix

Flash Attention 2 requires FP16 or BF16 KV cache — it cannot use quantized KV caches. If you're using vLLM with AWQ and Flash Attention, the weights are quantized but the KV cache stays in FP16, which limits the total memory savings. Check your engine's documentation for the specific combination you're using.

Pitfall 1: Assuming All Quants Are Created Equal

A Q4 from one engine is not the same as a Q4 from another. GGUF Q4_K_M uses block-wise quantization with super-blocks and outlier handling. Bitsandbytes 4-bit (NF4) uses a normal-float data type. AWQ uses activation-aware scaling. They all target ~4 bits but achieve different actual quality and speed. Don't compare quant types across engines without benchmarking.

Pitfall 2: 8-Bit ≠ Q8_0

Beginners often confuse "8-bit quantization" (INT8, Bitsandbytes) with Q8_0 in GGUF. They are different quantization schemes with different file formats. Q8_0 uses GGUF's block-wise approach; INT8 in vLLM or TGI uses per-tensor or per-channel scaling. They are not interchangeable.

Pitfall 3: Re-quantizing Already-Quantized Models

Never quantize an already-quantized model. Going from Q4_K_M → Q8_0 does not recover the lost information; it just wastes space. Always start from FP16 or BF16 when quantizing. If you only have access to quantized versions, download the original FP16 model from HuggingFace.

10. FAQ

Does quantization affect fine-tuned models differently?

Yes. Fine-tuned models (especially LoRA adapters) can be more sensitive to quantization because the fine-tuning process creates concentrated weight updates in specific layers. If you're quantizing a fine-tune, test aggressively — run your actual downstream tasks and compare results against FP16. Domain-specific fine-tunes on niche data are the most vulnerable.

Can I mix different quants across layers?

llama.cpp does this internally — Q4_K_M uses 4-bit for most weights but 6-bit for attention output projections. End users can't manually specify per-layer quants, but the scheme is designed so you don't need to. The distributions are tuned for each model architecture.

How much does quantization slow down prompt processing?

On GPU, prompt processing (prefill) is compute-bound, so lower quants can actually be faster — fewer bytes to decompress and process. On CPU, prompt processing is still memory-bandwidth bound, so it scales with bitrate like token generation. Typical prompt processing speeds for a 7B model on a mid-range GPU: 200–500 tok/s for Q4_K_M, 180–450 tok/s for Q8_0.

Is there a "lossless" quantization?

No. Any quantization that reduces precision discards information. The best you can do is "near-lossless" — Q8_0 and Q6_K achieve perplexity deltas under 0.5%, which is statistically indistinguishable from FP16 on most benchmarks. But the information is mathematically lost; you cannot recover the original FP16 weights from the quantized version.

What about FP8 and FP4 (new NVIDIA hardware)?

H100 and newer GPUs support native FP8 tensor operations. vLLM and TensorRT-LLM can leverage this for both weight and activation quantization with excellent throughput (2× over FP16). This is the future of quantization for datacenter GPUs, but it's not relevant for consumer hardware yet — no consumer GPU supports native FP8 compute.
