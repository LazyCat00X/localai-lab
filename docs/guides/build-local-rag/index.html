1. What is RAG and When to Use It

Retrieval-Augmented Generation (RAG) is an architecture that enhances large language model outputs by grounding generation on a private knowledge base. Instead of asking the LLM to recall facts from its training data — which may be outdated, incomplete, or nonexistent for your domain — you first retrieve the most relevant documents from a vector database, then inject them into the model's context window alongside the user's question.

RAG vs. Fine-Tuning vs. Pure Prompting

Approach Best For Trade-offs
Pure Prompting General knowledge Q&A, creative writing, code generation No knowledge of private data; limited by context window (4K–128K tokens); may hallucinate
Fine-Tuning Learning a new style, tone, or formatting; few-shot patterns repeated thousands of times Expensive (GPU hours); needs curated dataset; model becomes specialized at cost of general ability; doesn't add new factual knowledge reliably
RAG Factual Q&A over private docs, chat-with-your-data, customer support bots, research assistants Requires a retrieval pipeline; depends on chunk quality and embedding model; adds latency for retrieval; simplest to update (just swap documents)

The rule of thumb: if you need the model to know something it wasn't trained on (your company's internal docs, a specific PDF manual, a codebase), use RAG. If you need the model to behave differently (write in your brand voice, follow a strict output schema), consider fine-tuning. In practice, most teams start with RAG and only fine-tune later if they need behavioral changes that prompting alone can't achieve.

2. Architecture Overview

A local RAG system has five logical stages connected in a pipeline:

User Query
    │
    ▼
┌─────────────────────┐
│  Embedding Model    │  sentence-transformers (all-MiniLM-L6-v2)
│  (query → vector)   │
└────────┬────────────┘
         │ vector
         ▼
┌─────────────────────┐
│  Vector Database    │  ChromaDB (local, on-disk)
│  (cosine similarity)│
└────────┬────────────┘
         │ top-k chunks
         ▼
┌─────────────────────┐
│  LLM (Ollama)       │  llama3.2, mistral, phi4
│  (context + query)  │
└────────┬────────────┘
         │ answer
         ▼
      Response
  1. Embedding Model — converts text chunks and the user's query into dense vector representations. We use a local model via sentence-transformers so no data leaves your machine.
  2. Vector Database — stores document embeddings and supports similarity search. ChromaDB runs embedded in your Python process and persists to disk.
  3. Retrieval — given a query vector, finds the K most similar document vectors using cosine distance. Optional steps include MMR (Maximal Marginal Relevance) for diversity and hybrid search combining semantic + keyword matching.
  4. Context Injection — the retrieved text chunks are assembled into a prompt template that tells the LLM to answer using only the provided context.
  5. Generation — the prompt is sent to a local LLM running in Ollama, which produces the final answer grounded in your documents.

The beauty of this design is that every piece runs on your own hardware. No API keys, no data leaks, no per-token charges. A decent consumer GPU (6 GB VRAM or more) handles both the embedding model and the LLM comfortably.

3. System Setup

Before writing any code, install the required system and Python dependencies.

Install Ollama

Ollama makes it trivial to run open-source LLMs locally. Install it with a single command:

curl -fsSL https://ollama.com/install.sh | sh

Then pull a capable model. We recommend starting with mistral (7B) for a balance of quality and speed, or llama3.2:3b if you're on a laptop:

ollama pull mistral
# or ollama pull llama3.2:3b
# or ollama pull phi4:14b  (if you have 16 GB+ VRAM)

Verify Ollama is running with ollama list. The Ollama API will be available at http://localhost:11434.

Python Dependencies

Create a fresh virtual environment and install the required packages:

4. Document Ingestion

Ingestion is the offline pipeline that reads your documents, splits them into chunks, embeds each chunk, and stores everything in ChromaDB. You run this once (or whenever your document set changes).

Loading Documents

We'll support both plain text files and PDFs. The loader function detects the file extension and extracts text accordingly:

import os
from pypdf import PdfReader

def load_document(file_path: str) -> str:
    """Load text from a PDF or .txt file."""
    ext = os.path.splitext(file_path)[1].lower()
    if ext == ".pdf":
        reader = PdfReader(file_path)
        text = "\n".join(page.extract_text() for page in reader.pages)
        return text
    elif ext == ".txt":
        with open(file_path, "r", encoding="utf-8") as fh:
            return fh.read()
    else:
        raise ValueError(f"Unsupported file type: {ext}")

Chunking with Overlap

Raw text is too large to embed as a single vector (the embedding model has a maximum token limit of 512 for all-MiniLM-L6-v2). We split the text into overlapping chunks so that context isn't lost at boundaries:

import tiktoken

def chunk_text(text: str, chunk_size: int = 512,
               overlap: int = 64) -> list[dict]:
    """Split text into chunks of `chunk_size` tokens with `overlap`
    token overlap. Returns list of dicts with 'text' and 'index'."""
    enc = tiktoken.get_encoding("cl100k_base")
    tokens = enc.encode(text)
    chunks = []
    start = 0
    while start < len(tokens):
        end = min(start + chunk_size, len(tokens))
        chunk_tokens = tokens[start:end]
        chunk_text = enc.decode(chunk_tokens)
        chunks.append({
            "text": chunk_text,
            "index": len(chunks),
            "start_token": start,
            "end_token": end,
        })
        if end == len(tokens):
            break
        start += chunk_size - overlap  # slide window forward
    return chunks

The overlap parameter (typically 10–20% of chunk_size) ensures that a sentence or concept that straddles a chunk boundary is preserved in at least one of the two chunks. When the user later queries about a concept near the boundary, it will still be found.

Creating Embeddings and Storing in ChromaDB

import chromadb
from sentence_transformers import SentenceTransformer

# Initialize embedding model
embedder = SentenceTransformer("all-MiniLM-L6-v2")

# ChromaDB persists to disk at the given path
client = chromadb.PersistentClient(path="./chroma_data")
collection = client.get_or_create_collection(
    name="my_docs",
    metadata={"hnsw:space": "cosine"}  # cosine similarity
)

def ingest_document(file_path: str):
    """Load, chunk, embed, and store a document."""
    print(f"Ingesting: {file_path}")
    text = load_document(file_path)
    chunks = chunk_text(text, chunk_size=512, overlap=64)

    ids = []
    documents = []
    for chunk in chunks:
        doc_id = f"{os.path.basename(file_path)}#{chunk['index']}"
        ids.append(doc_id)
        documents.append(chunk["text"])

    # Batch embed — sentence-transformers handles GPU if available
    embeddings = embedder.encode(documents, show_progress_bar=True).tolist()

    # Store in ChromaDB
    collection.add(
        ids=ids,
        embeddings=embeddings,
        documents=documents,
        metadatas=[{"source": file_path, "chunk": c["index"]}
                   for c in chunks],
    )
    print(f"  → {len(chunks)} chunks stored.")

Usage is straightforward:

# Ingest a directory of documents
import glob
for path in glob.glob("./docs/**/*.pdf", recursive=True):
    ingest_document(path)
for path in glob.glob("./docs/**/*.txt", recursive=True):
    ingest_document(path)

You can re-run ingestion any time — ChromaDB deduplicates by ID. If you change your chunking strategy or embedding model, simply delete the chroma_data directory and re-ingest.

5. Chunking Strategies

Chunking is arguably the most impactful hyperparameter in a RAG pipeline. If chunks are too small, you lose context; if too large, you dilute relevance and exceed the embedding model's token limit. Here are the three main strategies:

Strategy How it Works Pros Cons Best For
Fixed-Size Split at exactly N tokens, optionally with overlap Simple, fast, deterministic; easy to reason about May split mid-sentence or mid-concept; loses semantic boundaries General-purpose; starting point for any project
Semantic Split at paragraph boundaries, section headers, or sentence ends Preserves logical units; better retrieval quality Variable chunk sizes; may produce very long chunks; more complex to implement Well-structured documents (manuals, Wikipedia, academic papers)
Recursive Try splitting on paragraph breaks, then sentences, then words, then characters (LangChain-style) Graceful fallback; always produces valid chunks Slightly slower; still may split semantic groups at deeper recursion levels Documents with mixed formatting; production pipelines

For most local RAG projects, fixed-size with 512 tokens and 64-token overlap is the recommended starting point. It's fast, the results are predictable, and with a good embedding model the retrieval quality is excellent. If you later find that answers are missing context, try increasing chunk_size to 768 or switching to semantic chunking using langchain.text_splitter.RecursiveCharacterTextSplitter.

6. Retrieval Methods

Once your documents are ingested, querying is a three-step process: embed the query, search the vector DB, and optionally re-rank or diversify the results.

Basic Cosine Similarity Search

def retrieve(query: str, k: int = 5) -> list[str]:
    """Retrieve top-k chunks for a query."""
    query_embedding = embedder.encode([query]).tolist()
    results = collection.query(
        query_embeddings=query_embedding,
        n_results=k,
    )
    return results["documents"][0]

This returns the K chunks with the highest cosine similarity to the query vector. For most use cases, K=5 is a good default — enough context for a thorough answer without overflowing the LLM's context window.

MMR (Maximal Marginal Relevance)

MMR diversifies results by re-ranking to balance relevance with novelty. ChromaDB supports this natively:

def retrieve_mmr(query: str, k: int = 5,
                 fetch_k: int = 20, lambda_param: float = 0.5) -> list[str]:
    """Retrieve with MMR diversity."""
    query_embedding = embedder.encode([query]).tolist()
    results = collection.query(
        query_embeddings=query_embedding,
        n_results=k,
        # ChromaDB MMR parameters
        lambda_mult=lambda_param,  # 0.5 = equal weight relevance/diversity
        include=["documents", "distances"],
    )
    # ChromaDB's query() with MMR needs collect-and-filter approach
    # Full MMR is available via collection.query(..., alpha=0.5)
    return results["documents"][0]

Set lambda_param close to 1.0 for pure relevance (no diversity) and close to 0.0 for maximum diversity. The sweet spot is 0.5–0.7 for most knowledge-base Q&A.

Hybrid Search (Semantic + Keyword)

Pure semantic search can miss exact phrase matches (e.g., model numbers, error codes). ChromaDB doesn't include built-in full-text search, but you can implement a simple hybrid by:

  1. Running the semantic retrieval above.
  2. Also filtering by keyword match on the $contains operator if the query contains known identifiers.
  3. Merging and de-duplicating both result sets.

For a production hybrid search, consider using Qdrant or Weaviate (both support local deployment with BM25 + vector search). But for most local RAG projects, pure cosine similarity with MMR is more than adequate.

7. Context Injection & Generation

Retrieval is useless without generation. The final step is to construct a prompt that includes the retrieved chunks as context, then call the local LLM via Ollama's API.

Prompt Template

The template instructs the model to answer based solely on the provided context. This reduces hallucination and keeps responses grounded:

RAG_PROMPT_TEMPLATE = """You are a helpful assistant that answers questions
using ONLY the context provided below. If the context does not contain the
information needed to answer the question, say "I don't have enough
information to answer that." Do not make up facts.

Context:
{context}

Question: {question}

Answer (based solely on the context above):"""

Generation via Ollama API

import requests
import json

OLLAMA_URL = "http://localhost:11434/api/generate"

def generate(context_chunks: list[str], question: str,
             model: str = "mistral") -> str:
    """Send context + question to Ollama and return the answer."""
    context = "\n\n---\n\n".join(context_chunks)
    prompt = RAG_PROMPT_TEMPLATE.format(
        context=context, question=question
    )

    payload = {
        "model": model,
        "prompt": prompt,
        "stream": False,
        "options": {
            "temperature": 0.2,  # low temp for factual answers
            "num_predict": 512,  # max output tokens
        },
    }
    resp = requests.post(OLLAMA_URL, json=payload, timeout=120)
    resp.raise_for_status()
    return resp.json()["response"]

Key parameters worth tuning:

  • temperature: 0.1–0.3 for factual Q&A (low randomness). Higher values (0.7–1.0) for creative or summarization tasks.
  • num_predict: maximum tokens in the response. 512 is generous for most answers; increase to 1024 if you need longer explanations.
  • top_k / top_p: leave at defaults for Q&A; these control nucleus sampling.

8. Complete Working Pipeline

Below is a full, save-able Python script that ties everything together. Save it as local_rag.py and run it against any directory of PDFs or text files.

#!/usr/bin/env python3
"""
local_rag.py — Complete Local RAG Pipeline
===========================================
Requires: pip install chromadb sentence-transformers pypdf requests tiktoken
Requires: Ollama running locally (ollama serve)

Usage:
    python local_rag.py ingest ./my_docs/
    python local_rag.py query "What is the refund policy?"
"""

import os
import sys
import glob
import json
import tiktoken
import requests
import chromadb
from pypdf import PdfReader
from chromadb.utils import embedding_functions
from sentence_transformers import SentenceTransformer

# ── Configuration ──────────────────────────────────────────────────
CHROMA_DIR = "./chroma_data"
COLLECTION_NAME = "my_docs"
CHUNK_SIZE = 512
CHUNK_OVERLAP = 64
EMBEDDING_MODEL = "all-MiniLM-L6-v2"
OLLAMA_URL = "http://localhost:11434/api/generate"
LLM_MODEL = "mistral"
TOP_K = 5

# ── Embedding Function ─────────────────────────────────────────────
embedder = SentenceTransformer(EMBEDDING_MODEL)

class LocalEmbeddingFunction(embedding_functions.EmbeddingFunction):
    """Adapter so ChromaDB can use our local sentence-transformers model."""
    def __init__(self, model):
        self.model = model
    def __call__(self, texts):
        return self.model.encode(texts).tolist()

client = chromadb.PersistentClient(path=CHROMA_DIR)
collection = client.get_or_create_collection(
    name=COLLECTION_NAME,
    embedding_function=LocalEmbeddingFunction(embedder),
    metadata={"hnsw:space": "cosine"},
)

# ── Helpers ────────────────────────────────────────────────────────

def load_document(file_path: str) -> str:
    ext = os.path.splitext(file_path)[1].lower()
    if ext == ".pdf":
        reader = PdfReader(file_path)
        return "\n".join(page.extract_text() for page in reader.pages)
    elif ext == ".txt":
        with open(file_path, "r", encoding="utf-8") as fh:
            return fh.read()
    else:
        raise ValueError(f"Unsupported: {ext}")

def chunk_text(text: str) -> list[dict]:
    enc = tiktoken.get_encoding("cl100k_base")
    tokens = enc.encode(text)
    chunks = []
    start = 0
    while start < len(tokens):
        end = min(start + CHUNK_SIZE, len(tokens))
        chunk_tokens = tokens[start:end]
        chunks.append({
            "text": enc.decode(chunk_tokens),
            "index": len(chunks),
            "start": start,
            "end": end,
        })
        if end == len(tokens):
            break
        start += CHUNK_SIZE - CHUNK_OVERLAP
    return chunks

# ── Commands ───────────────────────────────────────────────────────

def cmd_ingest(paths: list[str]):
    """Ingest all supported documents from given paths (files or dirs)."""
    files = []
    for p in paths:
        if os.path.isfile(p):
            files.append(p)
        else:
            files.extend(glob.glob(os.path.join(p, "**/*.pdf"), recursive=True))
            files.extend(glob.glob(os.path.join(p, "**/*.txt"), recursive=True))

    for fpath in files:
        print(f"Ingesting: {fpath}")
        text = load_document(fpath)
        chunks = chunk_text(text)
        ids = [f"{os.path.basename(fpath)}#{c['index']}" for c in chunks]
        documents = [c["text"] for c in chunks]
        metadatas = [{"source": fpath, "chunk": c["index"]} for c in chunks]

        # ChromaDB will use our LocalEmbeddingFunction automatically
        collection.add(ids=ids, documents=documents, metadatas=metadatas)
        print(f"  → {len(chunks)} chunks added.")

    print(f"\nTotal collection count: {collection.count()}")

def cmd_query(question: str):
    """Retrieve and generate an answer."""
    # Retrieve
    results = collection.query(
        query_texts=[question],
        n_results=TOP_K,
    )
    chunks = results["documents"][0]

    # Generate
    context = "\n\n---\n\n".join(chunks)
    prompt = f"""You are a helpful assistant. Answer using ONLY the context below.
If the context doesn't contain the answer, say you don't know.

Context:
{context}

Question: {question}

Answer (based only on the context):"""

    payload = {
        "model": LLM_MODEL,
        "prompt": prompt,
        "stream": False,
        "options": {"temperature": 0.2, "num_predict": 512},
    }
    resp = requests.post(OLLAMA_URL, json=payload, timeout=120)
    answer = resp.json()["response"]

    print(f"\n{'='*60}")
    print(f"Question: {question}")
    print(f"{'='*60}")
    print(f"\nAnswer:\n{answer}")
    print(f"\n{'='*60}")
    print(f"Retrieved from {len(chunks)} chunk(s):")
    for i, c in enumerate(chunks, 1):
        print(f"\n  [{i}] {c[:150]}..." if len(c) > 150 else f"\n  [{i}] {c}")

# ── CLI Entry Point ────────────────────────────────────────────────

if __name__ == "__main__":
    if len(sys.argv) < 2:
        print(__doc__)
        sys.exit(1)

    command = sys.argv[1]
    if command == "ingest":
        if len(sys.argv) < 3:
            print("Usage: python local_rag.py ingest  [path ...]")
            sys.exit(1)
        cmd_ingest(sys.argv[2:])
    elif command == "query":
        if len(sys.argv) < 3:
            print("Usage: python local_rag.py query \"your question here\"")
            sys.exit(1)
        cmd_query(" ".join(sys.argv[2:]))
    else:
        print(f"Unknown command: {command}")
        sys.exit(1)

This script is production-ready for personal or team use. Run it against your company's documentation, a textbook, or a manual and you'll immediately see the value: the LLM answers questions it could never have known from training alone.

Usage Example

# 1. Ingest all PDFs in ./docs/
python local_rag.py ingest ./docs/

# 2. Ask a question
python local_rag.py query "What are the system requirements for installation?"

9. Performance Tuning

Once your basic pipeline is working, you can optimize each component for better accuracy and speed.

Chunk Overlap

Overlap prevents concepts from being lost at chunk boundaries. With 512-token chunks, a 64-token overlap (12.5%) is a safe default. If you notice that the model is missing information that appears near chunk boundaries in your source documents, increase overlap to 128 tokens (25%). The trade-off is more chunks stored (roughly +15% storage) and slightly slower ingestion, but retrieval quality often improves noticeably.

Embedding Model Choices

Model Dimensions Speed (docs/sec on CPU) Quality (MTEB Score) Disk Size Use Case
all-MiniLM-L6-v2 384 ~800 60.0 90 MB Best all-rounder for local RAG; fast, small, good quality
all-mpnet-base-v2 768 ~200 63.3 440 MB Higher accuracy, slower; use if retrieval quality is critical
BAAI/bge-small-en-v1.5 384 ~900 61.1 33 MB Smallest footprint; great for resource-constrained devices
BAAI/bge-large-en-v1.5 1024 ~50 64.2 1.3 GB Best quality; needs GPU for reasonable throughput

Re-Ranking

A re-ranker is a small cross-encoder model that takes the query + each chunk and scores relevance more accurately than cosine similarity alone. The typical approach: retrieve top-20 with the embedding model, then re-rank to top-5 with a cross-encoder. This adds ~50–200 ms per query but significantly boosts accuracy for ambiguous queries.

# Optional re-ranking step
from sentence_transformers import CrossEncoder

reranker = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2")

def retrieve_with_rerank(query: str, k: int = 5,
                         fetch_k: int = 20) -> list[str]:
    """Retrieve fetch_k chunks, re-rank, return top-k."""
    query_embedding = embedder.encode([query]).tolist()
    results = collection.query(
        query_embeddings=query_embedding,
        n_results=fetch_k,
    )
    docs = results["document"][0] if "document" in results else results["documents"][0]

    pairs = [[query, doc] for doc in docs]
    scores = reranker.predict(pairs, show_progress_bar=False)

    # Sort by re-ranker score descending, return top-k
    scored = sorted(zip(docs, scores), key=lambda x: x[1], reverse=True)
    return [doc for doc, _ in scored[:k]]

Re-ranking is the single highest-leverage improvement you can make to retrieval quality. It's especially valuable when your vector DB returns chunks that are semantically similar but not actually relevant (e.g., both mention "installation" but only one describes the installation process).

Additional Optimization Tips

  • Batch ingestion: when ingesting hundreds of documents, batch chunks into groups of 100–500 before calling collection.add() for faster indexing.
  • Quantized embeddings: ChromaDB supports binary quantization (binary=True in collection metadata) which shrinks storage by 32× with minimal accuracy loss.
  • Parallel ingestion: use Python's concurrent.futures to load and chunk multiple documents simultaneously — I/O bound operations benefit greatly.
  • Cache embeddings: cache the embedding model's output so you only recompute embeddings for documents that have changed.
  • Monitor context window: ensure your context (retrieved chunks + prompt template + query) fits within the LLM's context limit. For Mistral 7B, that's 8K tokens; for llama3.2, 128K tokens. Adjust TOP_K accordingly.

10. Next Steps

You now have a fully functional, local, private RAG system. Here are some directions to explore next:

  • Add a web UI: build a simple Gradio or Streamlit interface so non-technical users can query the knowledge base through a chat interface.
  • Multi-modal RAG: extend the pipeline to handle images by using a vision-language model (e.g., LLaVA via Ollama) or by extracting text from images with OCR (Tesseract) before embedding.
  • Agentic RAG: give the LLM tools (search the web, run Python, call APIs) so it can fetch information dynamically rather than relying solely on the static vector DB.
  • Evaluation: build a test set of question-answer pairs from your documents and measure recall@k, MRR, and answer faithfulness. Tools like ragas or deepeval automate this.
  • Hybrid search with BM25: integrate Tantivy or the rank_bm25 package to add keyword-based retrieval alongside semantic search for better exact-match performance.
  • ChromaDB to Postgres with pgvector: if your knowledge base grows beyond 100K documents, migrate to Postgres with pgvector for ACID compliance, concurrent access, and full-text + vector hybrid queries.
