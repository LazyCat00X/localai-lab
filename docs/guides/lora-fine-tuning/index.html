Fine-tuning large language models used to require a fleet of A100s and deep pockets. That changed with LoRA (Low-Rank Adaptation) and QLoRA (Quantized LoRA). These techniques let you train a 7-billion-parameter model on a single consumer GPU — a $1,600 RTX 4090 with 24 GB of VRAM — and produce adapter weights that are just a few megabytes, yet rival full fine-tuning quality at 90–95% of the performance.

This guide is a hands-on, command-by-command walkthrough. You will learn what LoRA and QLoRA actually do, how much hardware you need for each model size, and three production-ready tools (Unsloth, Axolotl, and LLaMA-Factory) with real YAML configs and terminal output. By the end, you will have fine-tuned a model, merged the adapter weights, and served it — all from a Linux terminal.

What Are LoRA and QLoRA?

Full fine-tuning updates every weight in the model. A 7B-parameter model in FP16 requires roughly 14 GB just to store the weights, plus another 42 GB for optimizer states (Adam uses two momentum buffers) and ~14 GB for activations and gradients — totaling around 70 GB for a full fine-tune. That is four A100-40GB GPUs just to start training.

LoRA (Low-Rank Adaptation, Hu et al., 2021) freezes the original weights and injects small, trainable adapter matrices into the attention layers. Instead of updating the full 4096 × 4096 weight matrix in a Q or V projection, LoRA learns two low-rank matrices A and B of shape 4096 × r and r × 4096, where r (the LoRA rank) is a small number like 16 or 64. The result: you train roughly 0.1–1% of the total parameters, slashing VRAM by ~70% and cutting training time by 2–5× compared to full fine-tuning, with almost no quality degradation.

QLoRA (Dettmers et al., 2023) pushes this further by quantizing the frozen base model to 4-bit precision using the NF4 (NormalFloat4) data type and double quantization. The base model goes from 16-bit to 4-bit — a 4× memory compression — while the LoRA adapters remain in 16-bit (BF16/FP16). A 7B model that needs 14 GB in FP16 now fits in ~4 GB in 4-bit, leaving room for optimizer states and activations on a single 24 GB GPU. QLoRA typically achieves 80–90% of full fine-tuning quality, with some carefully tuned runs reaching 95%.

Key insight: LoRA and QLoRA are not fundamentally different algorithms. QLoRA is just LoRA applied on top of a 4-bit quantized base model. The adapter training math is identical — what changes is the precision of the frozen weights. When you have enough VRAM for FP16 LoRA (≥24 GB for 7B models), prefer it. When VRAM is tight, QLoRA is your ticket.
LoRA vs QLoRA vs Full Fine-Tune for a 7B Model
Method Trainable Params Min VRAM (7B) Adapter Size Quality vs Full FT Training Speed
Full Fine-Tune (FP16) 7B (100%) ~70 GB 14 GB (full model) 100% (baseline) 1×
LoRA (r=16, FP16) ~16M (0.23%) ~22 GB ~32 MB 90–95% 2–3× faster
QLoRA (r=16, 4-bit) ~16M (0.23%) ~10 GB ~32 MB 80–90% 1.5–2× faster
LoRA (r=64, FP16) ~64M (0.92%) ~26 GB ~128 MB 93–97% ~2× faster
QLoRA (r=64, 4-bit) ~64M (0.92%) ~14 GB ~128 MB 85–93% 1.5–2× faster

Note that QLoRA with r=64 on a 7B model runs comfortably on a single RTX 3090/4090 (24 GB). The 10–14 GB numbers above include room for a typical batch size of 4–8 and a 2048-token sequence length. With gradient checkpointing enabled, you can push sequence length to 4096 on the same hardware.

Dataset Preparation

All three tools accept datasets in standard formats. The most common are:

  • Alpaca format — JSON with instruction, input (optional), and output fields. Best for simple instruction-tuning.
  • ShareGPT format — JSON with conversations array of {"from": "human"/"gpt", "value": "..."} objects. Best for multi-turn chat fine-tuning.

Here is a minimal ShareGPT-format dataset saved to ~/datasets/my-chat-data.json:

[
  {
    "conversations": [
      {"from": "human", "value": "What is the capital of France?"},
      {"from": "gpt",   "value": "The capital of France is Paris."}
    ]
  },
  {
    "conversations": [
      {"from": "human", "value": "How do I check disk usage on Linux?"},
      {"from": "gpt",   "value": "Use the `df -h` command to see disk usage in human-readable format. For a specific directory: `du -sh /path/to/dir`."}
    ]
  },
  {
    "conversations": [
      {"from": "human", "value": "Write a Python function to reverse a string."},
      {"from": "gpt",   "value": "```python\ndef reverse_string(s: str) -> str:\n    return s[::-1]\n```"}
    ]
  }
]

How much data do you need? LoRA fine-tuning is sample-efficient. With 500–1,000 high-quality, diverse examples, you can teach a model a new style or domain-specific terminology. For deeper behavior changes (e.g., making a model follow a strict output format), aim for 2,000–5,000 examples. Beyond 10,000 examples, full fine-tuning may start to show advantages, though this is rarely necessary for domain adaptation.

Data quality over quantity: A dataset of 500 carefully curated, diverse, and correctly formatted examples will produce a better fine-tune than 5,000 noisy, repetitive ones. Deduplicate your data, remove examples over 4096 tokens (they get truncated anyway), and verify that every output is what you actually want the model to produce.

Fine-Tuning with Unsloth

Unsloth is the fastest way to LoRA/QLoRA fine-tune on a single GPU. It uses hand-written Triton and CUDA kernels that are 2× faster than Hugging Face's standard training loop and use 30–50% less VRAM. The workflow is a single Python script.

Installation

$ python -m venv unsloth-env
$ source unsloth-env/bin/activate
$ pip install unsloth
$ pip install datasets accelerate bitsandbytes

# Verify installation
$ python -c "from unsloth import FastLanguageModel; print('Unsloth ready')"
Unsloth ready

QLoRA Fine-Tuning Script

Save the following to train_unsloth.py. This script fine-tunes Llama 3.2-3B-Instruct on a custom chat dataset using QLoRA with rank 16, fitting in under 6 GB of VRAM:

import torch
from unsloth import FastLanguageModel
from datasets import load_dataset
from trl import SFTTrainer
from transformers import TrainingArguments

# ── 1. Load model with 4-bit QLoRA ────────────────────────────────
model, tokenizer = FastLanguageModel.from_pretrained(
    model_name="unsloth/Llama-3.2-3B-Instruct",
    max_seq_length=2048,
    dtype=None,          # auto-detect BF16/FP16
    load_in_4bit=True,   # QLoRA: 4-bit base model
)

# ── 2. Add LoRA adapters ──────────────────────────────────────────
model = FastLanguageModel.get_peft_model(
    model,
    r=16,                          # LoRA rank
    target_modules=[
        "q_proj", "k_proj", "v_proj", "o_proj",
        "gate_proj", "up_proj", "down_proj",
    ],
    lora_alpha=16,                 # scaling factor
    lora_dropout=0,                # 0 = optimized for performance
    bias="none",
    use_gradient_checkpointing="unsloth",  # 30% less VRAM
    random_state=3407,
)

# ── 3. Load and format dataset ────────────────────────────────────
dataset = load_dataset("json", data_files="~/datasets/my-chat-data.json",
                        split="train")

def format_chat(ex):
    """Convert ShareGPT to model's chat template."""
    return {"text": tokenizer.apply_chat_template(
        ex["conversations"], tokenize=False, add_generation_prompt=False
    )}

dataset = dataset.map(format_chat)

# ── 4. Train ──────────────────────────────────────────────────────
trainer = SFTTrainer(
    model=model,
    tokenizer=tokenizer,
    train_dataset=dataset,
    dataset_text_field="text",
    max_seq_length=2048,
    args=TrainingArguments(
        per_device_train_batch_size=2,
        gradient_accumulation_steps=4,    # effective batch = 8
        warmup_steps=5,
        num_train_epochs=3,
        learning_rate=2e-4,
        fp16=not torch.cuda.is_bf16_supported(),
        bf16=torch.cuda.is_bf16_supported(),
        logging_steps=1,
        optim="adamw_8bit",
        weight_decay=0.01,
        lr_scheduler_type="cosine",
        seed=3407,
        output_dir="outputs",
        report_to="none",
    ),
)

trainer.train()

# ── 5. Save adapter (LoRA weights only, ~32 MB) ──────────────────
model.save_pretrained("lora-adapter")
tokenizer.save_pretrained("lora-adapter")
print("Adapter saved to ./lora-adapter/")

Fine-Tuning with Axolotl

Axolotl is a YAML-config-driven fine-tuning framework. Instead of writing Python, you declare your model, dataset, adapter, and hyperparameters in a single YAML file and run axolotl train config.yml. It is the tool of choice for reproducible, production-grade fine-tuning and supports DeepSpeed, FSDP, and multi-node training out of the box.

Hyperparameter Deep Dive

Getting hyperparameters right is the difference between a model that nails your task and one that outputs gibberish. Here is what each knob does and how to tune it.

LoRA Rank (r) and Alpha

LoRA Rank Selection Guide
Rank (r) Trainable Params (7B) Adapter Size Use Case
8 ~8M (0.11%) ~16 MB Style transfer, minor behavior tweaks
16 ~16M (0.23%) ~32 MB Domain adaptation, instruction following (sweet spot)
32 ~33M (0.47%) ~66 MB Complex task learning, code generation
64 ~67M (0.93%) ~134 MB Deep domain expertise, multi-task fine-tuning
128 ~135M (1.86%) ~270 MB Near-full-fine-tune quality, large datasets

lora_alpha is a scaling factor applied to the LoRA output. The effective scaling is alpha / r. Common practice: set lora_alpha = 2 * lora_r (e.g., r=16, alpha=32), which gives a scaling factor of 2.0. Doubling alpha effectively doubles the learning rate for LoRA weights. Start with the 2x rule and only tune alpha if you see underfitting (increase alpha) or unstable loss (decrease alpha).

Learning Rate

LoRA fine-tuning is more forgiving than full fine-tuning. A learning rate of 2e-4 (AdamW) is the community default and works well for most cases. For QLoRA, you can often push to 5e-4 because the frozen 4-bit weights act as a regularizer. If your loss oscillates or spikes, halve the learning rate. If the loss plateaus after 1–2 epochs, try doubling it.

Batch Size and Gradient Accumulation

The effective batch size = per_device_train_batch_size × gradient_accumulation_steps. For LoRA fine-tuning, an effective batch of 8–32 works well. Smaller models (1–3B) tolerate larger batches; larger models (70B) need smaller batches due to memory constraints. Always use gradient accumulation if your GPU cannot fit the desired batch — a micro_batch of 1 with accumulation_steps=16 produces identical gradients to a batch of 16 (though it takes slightly longer due to kernel launch overhead).

Epochs and Early Stopping

LoRA adapters are highly expressive for their size, which means they will overfit if trained too long. For datasets under 5K examples, 3–5 epochs is typical. For 10K+, 1–3 epochs is usually enough. Always monitor the validation loss — when it stops decreasing (or starts increasing) while training loss keeps going down, stop training immediately. Axolotl's eval_steps and LLaMA-Factory's eval_strategy: steps automate this check.

Target Modules

The standard target modules for Llama/Mistral/Qwen architectures are all linear layers in the attention and MLP blocks:

target_modules = [
    "q_proj", "k_proj", "v_proj", "o_proj",   # attention
    "gate_proj", "up_proj", "down_proj",       # MLP
]

Adding lm_head and embed_tokens rarely improves quality and significantly increases adapter size. Stick with the attention + MLP projections — they capture all the expressive power you need.

Merging and Inference

After training, you have LoRA adapter weights (typically adapter_model.safetensors, 16–270 MB). You have two options for using them:

Option 1: Load adapter on top of base model (no merge). This keeps your disk usage low and lets you swap adapters at runtime. Most inference servers (vLLM, text-generation-inference) support this natively.

Option 2: Merge adapter into base model. This produces a single, self-contained model file that works with any inference engine, including Ollama, llama.cpp, and LM Studio. The merged model is the same size as the base model (no quality loss), just updated with your fine-tuned knowledge.

Merging with Unsloth

$ python -c "
from unsloth import FastLanguageModel

# Load base model + adapter
model, tokenizer = FastLanguageModel.from_pretrained(
    model_name='unsloth/Llama-3.2-3B-Instruct',
    adapter_name='./lora-adapter',
    load_in_4bit=False,          # merge requires FP16/BF16
    max_seq_length=2048,
)

# Merge and save
model.save_pretrained_merged(
    'llama-3.2-3b-my-finetune',
    tokenizer,
    save_method='merged_16bit',   # or 'merged_4bit_forced'
)
print('Merged model saved to ./llama-3.2-3b-my-finetune/')
"

Merging with Axolotl / Generic PEFT

$ python -c "
import torch
from transformers import AutoModelForCausalLM, AutoTokenizer
from peft import PeftModel

base = 'mistralai/Mistral-7B-Instruct-v0.3'
adapter = './mistral-7b-qlora-alpaca'

    base,
    torch_dtype=torch.bfloat16,

"

Troubleshooting

CUDA Out of Memory

If you hit OOM, try in this order: (1) enable gradient checkpointing (use_gradient_checkpointing="unsloth" or gradient_checkpointing: true); (2) reduce max_seq_length from 2048 to 1024; (3) reduce per_device_train_batch_size to 1 and increase gradient_accumulation_steps; (4) switch from LoRA (FP16) to QLoRA (4-bit). Each step recovers significant VRAM without degrading the final model quality.

Loss Not Decreasing

First, check that your dataset is actually loading — print a few tokenized examples. If data is fine, increase learning_rate from 2e-4 to 5e-4. If the loss still stalls, check that lora_alpha is set to at least 2 * lora_r. A very low alpha can silently kill training by scaling LoRA contributions to near zero.

Model Outputs Garbage After Fine-Tuning

This usually means overfitting. Check if validation loss diverged from training loss during training. Reduce num_train_epochs from 3 to 1 and try again. Also ensure you are not training on add_generation_prompt=True — the model should see the assistant's response during training, not generate it. The template should be applied with add_generation_prompt=False.

Flash Attention Not Working

Flash Attention requires a GPU with compute capability 8.0+ (Ampere or newer: A100, RTX 3090, RTX 4090). On older GPUs (V100, T4, RTX 2080), disable Flash Attention (flash_attention: false in Axolotl, or skip use_flash_attention_2=True) and rely on PyTorch's native SDPA attention, which is nearly as fast on supported hardware.

BF16 Not Supported

BF16 requires Ampere+ GPUs. If you see a BF16 error, set fp16: true and bf16: false. In Unsloth, the dtype=None auto-detection handles this. In Axolotl, use bf16: auto. Training in FP16 is functionally equivalent to BF16 for LoRA fine-tuning — the quality difference is negligible.

Next Steps

You have now fine-tuned a large language model with LoRA/QLoRA, merged the adapter, and verified it produces sensible output. Here is where to go next:

  • Evaluate your fine-tune: Run lm-eval-harness benchmarks (MMLU, GSM8K, HumanEval) on your fine-tuned model to quantify improvement over the base model.
  • Serve it with vLLM: Deploy your merged model using vLLM with LoRA hot-swapping for multi-tenant serving. Load multiple adapters and switch between them per-request.
  • Quantize for CPU inference: Convert your merged model to GGUF with llama.cpp's converter and run it on CPU with Ollama.
  • Try DPO alignment: After SFT fine-tuning, run Direct Preference Optimization (DPO) to align your model with human preferences. Axolotl and LLaMA-Factory support DPO out of the box.
  • Build a RAG system on top: Combine your fine-tuned model with a local RAG pipeline for grounded, up-to-date responses.
