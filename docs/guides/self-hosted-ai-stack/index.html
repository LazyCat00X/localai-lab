1. Architecture Overview

A production self-hosted AI stack is more than just running ollama serve on a spare machine. It's a layered system where each component solves a specific problem: model serving, user interface, traffic management, security, and observability. Here's the canonical architecture we'll build:

                          INTERNET
                             │
                    ┌────────▼────────┐
                    │   Nginx :443     │  ← TLS termination, rate limiting
                    │  (reverse proxy) │
                    └───┬─────────┬────┘
                        │         │
           ┌────────────▼──┐   ┌──▼──────────────┐
           │  Open WebUI    │   │  External API     │
           │  :8080 (web UI)│   │  Consumers       │
           └───────┬────────┘   └────────┬─────────┘
                   │                     │
         ┌─────────▼─────────────────────▼──────┐
         │         Ollama :11434                 │
         │    (general-purpose inference)         │
         └──────────────┬────────────────────────┘
                        │
         ┌──────────────▼────────────────────────┐
         │         vLLM :8000                     │
         │    (high-throughput inference)          │
         └──────────────┬────────────────────────┘
                        │
         ┌──────────────▼────────────────────────┐
         │    NVIDIA GPU(s) + CUDA 12.x           │
         │    (shared via nvidia-container-toolkit)│
         └───────────────────────────────────────┘

This is not a theoretical diagram. Every arrow represents a real TCP connection, every port a real bind address. The stack handles three traffic flows:

  1. Web UI traffic: Browser → Nginx → Open WebUI → Ollama/vLLM → GPU
  2. API traffic: External client → Nginx → Ollama/vLLM (OpenAI-compatible endpoint) → GPU
  3. Management traffic: Admin → SSH → host → Docker socket / systemd
Why two inference engines? Ollama handles the "long tail" — smaller models, experimentation, quick prototyping. vLLM handles production workloads where throughput matters (50+ concurrent requests, continuous batching). Both share the same GPU pool; you manage which model runs where based on workload characteristics.

2. Inference Engine Selection

You don't pick one inference engine. You pick the right one for each workload. The stack runs multiple engines side by side on the same hardware, fronted by a unified API layer. Here's the decision matrix:

When to Run Both

The most common production pattern: vLLM for your primary model, Ollama for everything else.

# vLLM: High-throughput production endpoint (Llama 3.1 70B on 2×A100)
$ vllm serve meta-llama/Llama-3.1-70B-Instruct \
    --tensor-parallel-size 2 \
    --max-model-len 8192 \
    --gpu-memory-utilization 0.90 \
    --port 8000

# Ollama: Quick experimentation, smaller models
$ OLLAMA_HOST=0.0.0.0:11434 ollama serve
$ ollama pull mistral:7b
$ ollama pull nomic-embed-text    # for RAG embeddings
GPU memory partitioning: On a single 48 GB A6000, you can run vLLM with Llama 3.1 8B (~18 GB) and Ollama with Mistral 7B Q4 (~4.5 GB) simultaneously. Total: ~22.5 GB VRAM used, leaving headroom for KV cache and system overhead. See Section 5 for the math.

3. Nginx Reverse Proxy Configuration

Never expose Ollama or vLLM directly to the internet. Both lack authentication and rate limiting. Nginx sits in front as the single entry point, handling TLS, routing, and security. Here's the production config:

# /etc/nginx/sites-available/localai-lab
upstream ollama_backend {
    server 127.0.0.1:11434;
    keepalive 32;
}

upstream vllm_backend {
    server 127.0.0.1:8000;
    keepalive 32;
}

upstream openwebui_backend {
    server 127.0.0.1:8080;
    keepalive 16;
}

server {
    listen 443 ssl http2;
    server_name ai.yourdomain.com;

    ssl_certificate     /etc/letsencrypt/live/ai.yourdomain.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/ai.yourdomain.com/privkey.pem;
    ssl_protocols TLSv1.3;

    # Rate limiting: 60 requests/min per IP
    limit_req_zone $binary_remote_addr zone=api_limit:10m rate=60r/m;
    limit_req_status 429;

    # ─── Ollama API (OpenAI-compatible) ───
    location /api/ollama/ {
        limit_req zone=api_limit burst=10 nodelay;
        rewrite ^/api/ollama/(.*) /$1 break;
        proxy_pass http://ollama_backend;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_buffering off;              # Critical for streaming
        proxy_read_timeout 300s;          # Long generation timeout
        proxy_set_header Connection '';
    }

    # ─── vLLM API ───
    location /api/vllm/ {
        limit_req zone=api_limit burst=20 nodelay;
        rewrite ^/api/vllm/(.*) /$1 break;
        proxy_pass http://vllm_backend;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_buffering off;
        proxy_read_timeout 300s;
        proxy_set_header Connection '';
    }

    # ─── Open WebUI ───
    location / {
        proxy_pass http://openwebui_backend;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";   # WebSocket support
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_read_timeout 86400s;               # Long-lived WebSocket
    }
}

Key Configuration Decisions

SettingValueWhy
proxy_bufferingoffLLM responses stream token-by-token. Buffering would delay every token until the full response is generated — destroying the real-time UX.
proxy_read_timeout300sLong generations (e.g., 8K output tokens) can take minutes. Default 60s timeout kills mid-generation.
limit_req60r/m per IPPrevents a single user from saturating GPU capacity. Adjust based on your hardware — 60 r/m is conservative for a single GPU.
keepalive 3232 connectionsMaintains persistent connections to backends. Each new TCP handshake adds ~2ms latency; keepalive eliminates this overhead.
Separate upstream blocksollama, vllm, openwebuiIndependent connection pools. A flood of Ollama requests shouldn't starve vLLM connections.
Streaming is non-negotiable: Both proxy_buffering off and proxy_http_version 1.1 are required for SSE (Server-Sent Events) streaming. HTTP/1.0 doesn't support chunked transfer encoding properly. If you see tokens arriving in bursts instead of streaming, check these two settings first.

4. The Frontend Layer

Open WebUI is the de facto standard for self-hosted LLM chat interfaces. It provides user management, conversation history, RAG document upload, and model switching — everything you'd expect from a ChatGPT clone, but running on your hardware. Here's how it fits into the stack:

$ docker run -d \
    --name open-webui \
    --network host \
    -v open-webui:/app/backend/data \
    -e OLLAMA_BASE_URL=http://127.0.0.1:11434 \
    -e WEBUI_SECRET_KEY=$(openssl rand -hex 32) \
    ghcr.io/open-webui/open-webui:main

Multi-Model Routing

Open WebUI can talk to multiple backends simultaneously. Configure model-to-backend mappings under Admin Settings → Connections:

# Open WebUI model routing config (set via Admin UI or API)
OLLAMA_BASE_URL=http://127.0.0.1:11434    # Default
OPENAI_API_BASE_URL=http://127.0.0.1:8000/v1  # vLLM as "OpenAI" backend

This lets users switch between Ollama-hosted Mistral for quick chats and vLLM-hosted Llama 3.1 70B for complex reasoning — all from the same UI, with transparent routing.

5. GPU Management & Resource Allocation

The GPU is the scarcest resource in the stack. Every component competes for VRAM. Here's how to model and manage it:

VRAM Budgeting (RTX 4090 — 24 GB Example)

ComponentVRAM UsageNotes
vLLM: Llama 3.1 8B (FP16)~16 GBModel weights + KV cache
Ollama: Mistral 7B (Q4_K_M)~4.5 GBQuantized GGUF, always loaded
Ollama: nomic-embed-text~0.3 GBFor RAG embeddings
CUDA context overhead~0.5 GBDriver + CUDA runtime
System headroom~2 GBBuffer for KV cache spikes
Total~23.3 GBFits in 24 GB with 0.7 GB spare
Golden rule: Never configure --gpu-memory-utilization above 0.90 in vLLM. The remaining 10% is needed for CUDA graphs, temporary buffers during prefill, and kernel launches. Pushing to 0.95 causes OOM errors under concurrent load, not at startup — the worst kind, because they happen at 3 AM when your API has actual traffic.

6. Docker Compose Orchestration

The full stack runs as a set of Docker containers, orchestrated via Compose. This gives you reproducible deployments, health checks, automatic restarts, and resource isolation. Here's the complete production compose file:

# docker-compose.yml
version: '3.8'

services:
  # ─── Ollama ───
  ollama:
    image: ollama/ollama:0.5.12
    container_name: ollama
    runtime: nvidia
    environment:
      - NVIDIA_VISIBLE_DEVICES=all
      - OLLAMA_HOST=0.0.0.0
      - OLLAMA_KEEP_ALIVE=5m
    ports:
      - "127.0.0.1:11434:11434"
    volumes:
      - ollama_data:/root/.ollama
    deploy:
      resources:
        reservations:
          devices:
            - driver: nvidia
              count: 1
              capabilities: [gpu]
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "ollama", "list"]
      interval: 30s
      timeout: 10s
      retries: 3

  # ─── vLLM ───
  vllm:
    image: vllm/vllm-openai:v0.6.6
    container_name: vllm
    runtime: nvidia
    environment:
      - NVIDIA_VISIBLE_DEVICES=all
    command: >
      --model meta-llama/Llama-3.1-8B-Instruct
      --max-model-len 8192
      --gpu-memory-utilization 0.85
      --port 8000
    ports:
      - "127.0.0.1:8000:8000"
    volumes:
      - hf_cache:/root/.cache/huggingface
    deploy:
      resources:
        reservations:
          devices:
            - driver: nvidia
              count: 1
              capabilities: [gpu]
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8000/health"]
      interval: 30s
      timeout: 10s
      retries: 5
      start_period: 120s

  # ─── Open WebUI ───
  open-webui:
    image: ghcr.io/open-webui/open-webui:main
    container_name: open-webui
    network_mode: host
    environment:
      - OLLAMA_BASE_URL=http://127.0.0.1:11434
      - WEBUI_SECRET_KEY=${WEBUI_SECRET_KEY}
      - WEBUI_AUTH=True
    volumes:
      - webui_data:/app/backend/data
    restart: unless-stopped
    depends_on:
      ollama:
        condition: service_healthy

  # ─── Nginx ───
  nginx:
    image: nginx:1.27-alpine
    container_name: nginx
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./nginx.conf:/etc/nginx/conf.d/default.conf:ro
      - /etc/letsencrypt:/etc/letsencrypt:ro
      - nginx_logs:/var/log/nginx
    restart: unless-stopped
    depends_on:
      - ollama
      - vllm
      - open-webui

  # ─── Prometheus (monitoring) ───
  prometheus:
    image: prom/prometheus:v2.54.0
    container_name: prometheus
    ports:
      - "127.0.0.1:9090:9090"
    volumes:
      - ./prometheus.yml:/etc/prometheus/prometheus.yml:ro
      - prometheus_data:/prometheus
    command:
      - '--config.file=/etc/prometheus/prometheus.yml'
      - '--storage.tsdb.path=/prometheus'
      - '--storage.tsdb.retention.time=30d'
    restart: unless-stopped

  # ─── Grafana (dashboards) ───
  grafana:
    image: grafana/grafana:11.2.0
    container_name: grafana
    ports:
      - "127.0.0.1:3000:3000"
    volumes:
      - grafana_data:/var/lib/grafana
    environment:
      - GF_SECURITY_ADMIN_PASSWORD=${GRAFANA_PASSWORD}
      - GF_INSTALL_PLUGINS=grafana-piechart-panel
    restart: unless-stopped

volumes:

Health Check Strategy

ServiceCheckStart PeriodWhy
Ollamaollama listNoneStarts in <2s, simple process check
vLLMGET /health120sModel loading + warmup takes 30–90s. Without start_period, Compose kills it as unhealthy.
Open WebUIdepends_on ollama—No built-in health endpoint; rely on restart policy
Nginxdepends_on backends—Starts instantly; backends checked via depends_on
Pitfall: vLLM health check race condition. vLLM returns 200 on /health only after the model is fully loaded into VRAM and ready to serve. On a 70B model across 2 GPUs, this can take 90+ seconds. Set start_period: 120s or Docker will kill the container before it's ready. Also ensure restart: unless-stopped — if vLLM OOMs under load, it should restart automatically.

7. Monitoring & Observability

A self-hosted AI stack without monitoring is a black box. When your API latency spikes from 200ms to 8 seconds, you need to know whether it's GPU memory pressure, request queuing, or a model loading cold. Here's the monitoring stack:

Prometheus Metrics to Track

# prometheus.yml
global:
  scrape_interval: 15s

scrape_configs:
  # NVIDIA GPU metrics via nvidia-dcgm-exporter
  - job_name: 'dcgm'
    static_configs:
      - targets: ['localhost:9400']

  # vLLM built-in metrics
  - job_name: 'vllm'
    static_configs:
      - targets: ['localhost:8000']
    metrics_path: '/metrics'

  # Node exporter (CPU, RAM, disk)
  - job_name: 'node'
    static_configs:
      - targets: ['localhost:9100']

Don't skip GPU temperature monitoring. An RTX 4090 at 85°C under sustained load will thermally throttle, cutting throughput by 20–40% without any visible error. The only symptom is slower response times. Set a Grafana alert at 80°C — if you hit it, improve case airflow before buying another GPU.

8. Security Hardening

Self-hosted doesn't mean "secure by default." Here's the layered security model for a production stack:

Layer 1: Network Isolation

# All backends bind to localhost only — nothing exposed to the network
# Docker Compose: use "127.0.0.1:PORT:PORT" not "PORT:PORT"
ports:
  - "127.0.0.1:11434:11434"   # Ollama: localhost only
  - "127.0.0.1:8000:8000"    # vLLM: localhost only
  - "127.0.0.1:8080:8080"    # Open WebUI: localhost only

# Only Nginx binds to all interfaces
ports:
  - "443:443"                  # Public HTTPS endpoint

Layer 2: Authentication

ComponentAuth MethodConfiguration
Open WebUIBuilt-in user accountsWEBUI_AUTH=True + WEBUI_SECRET_KEY set
Ollama APINginx basic auth or API keyproxy_set_header Authorization with bearer token validation
vLLM APIAPI key via --api-keyvllm serve ... --api-key sk-prod-xxxx
GrafanaPassword + optional OAuthGF_SECURITY_ADMIN_PASSWORD

Layer 3: TLS Everywhere

# Certbot for Let's Encrypt auto-renewal
$ sudo apt install certbot python3-certbot-nginx
$ sudo certbot --nginx -d ai.yourdomain.com

# Auto-renewal cron (certbot adds this automatically)
$ cat /etc/cron.d/certbot
0 */12 * * * root certbot -q renew --deploy-hook "docker exec nginx nginx -s reload"

Layer 4: Firewall

$ sudo ufw default deny incoming
$ sudo ufw default allow outgoing
$ sudo ufw allow ssh
$ sudo ufw allow 443/tcp
$ sudo ufw enable

# Verify only intended ports are open
$ sudo ufw status verbose
Status: active
To                         Action      From
--                         ------      ----
22/tcp                     ALLOW       Anywhere
443/tcp                    ALLOW       Anywhere
Critical: Ollama has zero built-in authentication. If you bind Ollama to 0.0.0.0 without a reverse proxy, anyone who can reach port 11434 can run models on your GPU. This is not theoretical — Shodan regularly indexes exposed Ollama instances. Always put Nginx (with auth) in front.

9. Scaling Strategies

The stack scales in three dimensions: up (bigger GPU), out (more GPUs), and across (multiple nodes). Here's when each makes sense:

Scale Across: Load-Balanced Multi-Node

# Nginx upstream with multiple vLLM instances (different machines)
upstream vllm_cluster {
    least_conn;    # Distribute to instance with fewest active connections
    server 10.0.1.10:8000 weight=1 max_fails=3 fail_timeout=30s;
    server 10.0.1.11:8000 weight=1 max_fails=3 fail_timeout=30s;
    server 10.0.1.12:8000 weight=2 max_fails=3 fail_timeout=30s;  # A100 node
    keepalive 64;
}
StrategyComplexityCostWhen to Use
Scale upLow$$$ (single expensive GPU)1–10 users, single model

Start small, scale when you need it. A single RTX 4090 running Ollama handles 5–10 concurrent chat users comfortably. Only introduce vLLM when you hit queuing (users waiting >5s for first token). Only introduce multi-node when a single machine can't fit the model. Premature scaling adds complexity without benefit.

10. Backup & Recovery

Your stack's state lives in three places. Back up all of them:

WhatWhereBackup MethodFrequency
Model weights/root/.ollama/models/ or HF cacheDon't back up — re-download. Models are immutable and fetchable.Never
Open WebUI datawebui_data volumetar czf or rclone to S3Daily
Config filesdocker-compose.yml, nginx.conf, prometheus.ymlGit repo (already done if you're following this guide)On every change
SSL certificates/etc/letsencrypt/Let's Encrypt auto-renews; back up the account keyWeekly
 
# Daily backup script (add to crontab)
#!/bin/bash
BACKUP_DIR=/mnt/backups/localai-lab/$(date +%Y-%m-%d)
mkdir -p $BACKUP_DIR

# Open WebUI data
docker run --rm -v webui_data:/data -v $BACKUP_DIR:/backup \
    alpine tar czf /backup/webui-data.tar.gz -C /data .

# Config files (already in git, but snapshot anyway)
cp docker-compose.yml nginx.conf $BACKUP_DIR/

# Retention: keep 7 daily backups
find /mnt/backups/localai-lab/ -maxdepth 1 -type d -mtime +7 -exec rm -rf {} \;

11. Troubleshooting

Common Failure Modes

SymptomLikely CauseDiagnosisFix
vLLM container keeps restarting OOM during model load docker logs vllm 2>&1 | grep -i "out of memory" Lower --gpu-memory-utilization or reduce --max-model-len
Tokens arrive in bursts, not streaming Nginx buffering enabled curl -N http://localhost:8000/v1/chat/completions — if local is smooth but via Nginx is bursty Verify proxy_buffering off in nginx.conf
Open WebUI shows "Ollama not connected" Ollama binding to 127.0.0.1, Docker can't reach docker exec open-webui curl http://host.docker.internal:11434 Use network_mode: host or OLLAMA_BASE_URL=http://host.docker.internal:11434
