What is vLLM?

vLLM is an open-source high-throughput LLM inference engine developed at UC Berkeley. It is designed to serve large language models with minimal latency and maximal throughput, making it the go-to choice for production AI serving. Unlike naive inference implementations that waste GPU memory on pre-allocated KV-cache slots, vLLM introduces PagedAttention — a memory management technique inspired by operating system virtual memory paging.

PagedAttention works by dividing the KV-cache into fixed-size blocks (pages) that are allocated on demand. This eliminates internal fragmentation and allows the system to pack more concurrent requests into the same GPU memory. On top of that, vLLM implements continuous batching: instead of waiting for a full batch to finish before accepting new requests, the engine dynamically adds and removes sequences from the active batch after every decoding step. The result is dramatically higher GPU utilization and lower time-to-first-token (TTFT).

Key features at a glance:

  • PagedAttention for near-zero KV-cache waste
  • Continuous batching for real-time request handling
  • OpenAI-compatible REST API out of the box
  • Tensor parallelism across multiple GPUs
  • Support for AWQ, GPTQ, and FP8 quantization
  • LoRA adapter hot-swapping at runtime
  • Prefix caching for shared prompt prefixes
  • Speculative decoding for latency reduction

vLLM supports a wide range of model architectures including Llama, Mistral, Mixtral, Qwen, DeepSeek, Gemma, Phi, Falcon, GPT-NeoX, and many more. It integrates natively with Hugging Face Transformers and can load models from the hub with a single line of code.

Installation

vLLM can be installed via pip, from source (for custom CUDA kernels), or via Docker for production deployments. Below are all three methods.

Option 1: pip Install (Recommended)

The simplest way to get started is with pip. vLLM ships pre-built wheels for Linux on x86_64 with CUDA 11.8, 12.1, and 12.4.

$ python -m venv vllm-env
$ source vllm-env/bin/activate
$ pip install vllm

# Verify the installation
$ python -c "import vllm; print(vllm.__version__)"

On a fresh Ubuntu 22.04 / 24.04 system, you may need to install CUDA drivers and PyTorch first:

$ pip install torch --index-url https://download.pytorch.org/whl/cu124
$ pip install vllm

Option 2: Build from Source

Building from source is useful if you need custom CUDA architecture flags, or if you are on a platform without pre-built wheels (e.g., ARM, ROCm, or custom CUDA versions). You need a CUDA toolkit (12.x) and PyTorch installed.

$ git clone https://github.com/vllm-project/vllm.git
$ cd vllm
$ pip install -e .  # editable install, compiles custom kernels

# For AMD ROCm:
# export VLLM_TARGET_DEVICE=rocm && pip install -e .

# For specific GPU architectures (faster compile):
# export TORCH_CUDA_ARCH_LIST="8.0;8.6;8.9;9.0"

Building from source can take 10–30 minutes depending on your CPU. The compilation produces optimized CUDA kernels for PagedAttention and the custom sampling ops.

Option 3: Docker

The official vLLM Docker image is the best option for production. It comes pre-compiled and optimized.

$ docker pull vllm/vllm-openai:latest

# Quick smoke-test
$ docker run --rm --gpus all \
    -p 8000:8000 \
    vllm/vllm-openai:latest \
    --model meta-llama/Meta-Llama-3.1-8B-Instruct

The Docker image includes all CUDA runtime dependencies, avoids host library conflicts, and can be used directly with Kubernetes or any container orchestrator.

Basic Usage

Once vLLM is installed, you can load a model and run inference using the Python API. Here is a minimal example:

from vllm import LLM, SamplingParams

# Load the model — this downloads from Hugging Face if not cached
llm = LLM(
    model="meta-llama/Meta-Llama-3.1-8B-Instruct",
    tensor_parallel_size=1,      # set to number of GPUs
    gpu_memory_utilization=0.90, # leave headroom for CUDA kernels
    max_model_len=8192,          # context window
)

# Configure generation parameters
sampling_params = SamplingParams(
    temperature=0.7,
    top_p=0.9,
    max_tokens=512,
    stop=["<|eot_id|>", "<|end_of_text|>"],
)

# Run inference
prompts = [
    "Explain quantum computing in one paragraph.",
    "Write a Python function to merge two sorted lists.",
]
outputs = llm.generate(prompts, sampling_params)

# Print results
for output in outputs:
    prompt = output.prompt
    generated = output.outputs[0].text
    print(f"Prompt: {prompt}")
    print(f"Generated: {generated}")
    print("-" * 60)

The LLM constructor handles model loading, weight quantization (if specified), and memory allocation. Each call to generate() processes the prompts concurrently using continuous batching — the engine will automatically schedule them for maximum throughput.

Streaming Output

For interactive applications, use streaming:

for output in llm.generate(prompts, sampling_params, use_tqdm=False):
    for req_id, stream in output:
        for chunk in stream:
            print(chunk.outputs[0].text, end="", flush=True)
        print()

OpenAI-Compatible API Server

vLLM ships a built-in OpenAI-compatible HTTP server. This means you can use any OpenAI SDK client (like openai-python) to talk to your vLLM instance. Start the server with:

$ python -m vllm.entrypoints.openai.api_server \
    --model meta-llama/Meta-Llama-3.1-8B-Instruct \
    --host 0.0.0.0 \
    --port 8000 \
    --tensor-parallel-size 1 \
    --max-model-len 8192 \
    --gpu-memory-utilization 0.90 \
    --served-model-name llama-3.1-8b

Then use it like OpenAI's API:

from openai import OpenAI

client = OpenAI(
    base_url="http://localhost:8000/v1",
    api_key="not-needed",  # vLLM accepts any string
)

response = client.chat.completions.create(
    model="llama-3.1-8b",
    messages=[
        {"role": "system", "content": "You are a helpful assistant."},
        {"role": "user", "content": "What is the capital of France?"},
    ],
    temperature=0.7,
    max_tokens=256,
)
print(response.choices[0].message.content)

The server exposes /v1/chat/completions, /v1/completions, /v1/embeddings, and /v1/models — fully compatible with the OpenAI API schema. This makes it a drop-in replacement for any application already using the OpenAI API.

Advanced: Quantization & LoRA

AWQ and GPTQ Quantization

vLLM supports AWQ (Adaptive Weight Quantization) and GPTQ (GPT Post-Training Quantization) to reduce model memory footprint by 2–4x with minimal accuracy loss. Loading an AWQ-quantized model is as simple as specifying the quantized hub path:

llm = LLM(
    model="TheBloke/Meta-Llama-3.1-8B-Instruct-AWQ",
    quantization="awq",
    gpu_memory_utilization=0.95,
)

For GPTQ:

llm = LLM(
    model="TheBloke/Llama-2-7B-Chat-GPTQ",
    quantization="gptq",
)

Quantization is handled at the kernel level — vLLM uses fused AWQ/GPTQ kernels that dequantize weights on-the-fly during the matmul, keeping arithmetic in FP16 while storing weights in INT4. The throughput difference vs FP16 is often within 5–10%, while memory usage drops by ~60%.

FP8 (H100/H200 Only)

On NVIDIA Hopper GPUs (H100, H200), vLLM supports FP8 quantization using the Transformer Engine. Enable it with --quantization fp8 or in Python:

llm = LLM(
    model="meta-llama/Meta-Llama-3.1-8B-Instruct",
    quantization="fp8",
    kv_cache_dtype="fp8",
)

FP8 halves both memory and bandwidth usage compared to FP16, yielding up to 2x throughput on H100 with comparable quality.

LoRA Adapters

vLLM supports serving multiple LoRA adapters simultaneously with zero-overhead switching. This is useful for multi-tenant setups where each user has a fine-tuned adapter.

from vllm import LLM, SamplingParams

llm = LLM(
    model="meta-llama/Meta-Llama-3.1-8B-Instruct",
    enable_lora=True,
    max_lora_rank=64,
    max_num_seqs=16,
)

# Generate with a specific LoRA adapter (by Hugging Face ID or local path)
outputs = llm.generate(
    prompts=["Explain the plot of Dune."],
    sampling_params=SamplingParams(temperature=0.8, max_tokens=256),
    lora_request="path/to/your-lora-adapter",
)

You can serve multiple LoRA adapters from the API server using the --lora-modules flag:

$ python -m vllm.entrypoints.openai.api_server \
    --model meta-llama/Meta-Llama-3.1-8B-Instruct \
    --enable-lora \
    --lora-modules lora-adapter-1=/path/to/lora1 lora-adapter-2=/path/to/lora2 \
    --max-lora-rank 64

The client selects the adapter by passing the model field as the LoRA module name in the API request.

Tensor Parallelism

For models too large for a single GPU (e.g., 70B or 405B), use tensor parallelism to split layers across GPUs:

$ python -m vllm.entrypoints.openai.api_server \
    --model meta-llama/Meta-Llama-3.1-405B-Instruct \
    --tensor-parallel-size 8 \
    --pipeline-parallel-size 1 \
    --max-model-len 16384 \
    --gpu-memory-utilization 0.95

Tensor parallelism shards each transformer layer's weights across GPUs and uses all-reduce for intermediate activations. vLLM uses custom CUDA kernels that minimize communication overhead by overlapping computation with NCCL transfers.

Performance Tuning

Getting the best throughput from vLLM requires tuning several knobs. The table below shows the most impactful parameters.

Parameter Default Recommended Notes
max_num_seqs 256 64–1024 Max concurrent sequences in a batch. Lower = lower latency, higher = higher throughput. Monitor GPU memory.
max_model_len model default 4096–32768 Context window size. Larger values consume more KV-cache memory per sequence.
gpu_memory_utilization 0.90 0.85–0.95 Fraction of GPU memory for the model + KV-cache. Leave headroom for CUDA kernels.
block_size 16 16–128 PagedAttention block size in tokens. Larger blocks reduce memory overhead but increase internal fragmentation.
swap_space 4 GB 4–16 GB CPU memory for offloading KV-cache blocks under memory pressure.
enable_prefix_caching False True (if shared prefix) Caches KV-blocks for common prompt prefixes. Huge win for few-shot or system-prompt-heavy workloads.

Latency vs. Throughput Tradeoff

For latency-sensitive applications (chatbots), reduce max_num_seqs to 16–64 and use --block-size 16. For throughput-heavy workloads (batch processing, RAG), increase max_num_seqs to 256+ and set --block-size 32 or 64. Use the --speculative-model flag to enable speculative decoding for an additional 1.5–2x speedup on latency.

Monitoring Performance

Enable vLLM's built-in stats logging to understand what's happening under the hood:

$ python -m vllm.entrypoints.openai.api_server \
    --model meta-llama/Meta-Llama-3.1-8B-Instruct \
    --disable-log-requests \
    --enable-prefix-caching \
    2>&1 | grep "avg prompt throughput"

The logs emit per-iteration metrics including average prompt throughput, generation throughput, number of running sequences, and GPU cache usage. Pipe these to a monitoring system like Prometheus for trend analysis.

Production Deployment

Docker Compose Setup

A production-ready Docker Compose configuration with health checks and auto-restart:

version: "3.9"
services:
  vllm:
    image: vllm/vllm-openai:latest
    container_name: vllm-server
    restart: unless-stopped
    ports:
      - "8000:8000"
    environment:
      - HF_TOKEN=${HF_TOKEN}    # for gated models
    volumes:
      - ~/.cache/huggingface:/root/.cache/huggingface
    command:
      - "--model"
      - "mistralai/Mistral-7B-Instruct-v0.3"
      - "--tensor-parallel-size"
      - "2"
      - "--gpu-memory-utilization"
      - "0.92"
      - "--max-model-len"
      - "16384"
      - "--max-num-seqs"
      - "256"
      - "--enable-prefix-caching"
    deploy:
      resources:
        reservations:
          devices:
            - driver: nvidia
              count: 2
              capabilities: [gpu]
    healthcheck:
      test: curl -f http://localhost:8000/health || exit 1
      interval: 30s
      timeout: 10s
      retries: 3

Multi-GPU and Multi-Node

vLLM supports both single-node multi-GPU (tensor parallelism) and multi-node (pipeline parallelism + tensor parallelism). For multi-node, set the VLLM_HOST and VLLM_PORT environment variables and launch with --pipeline-parallel-size N. vLLM uses NCCL for inter-node communication and automatically discovers peers via a shared filesystem or environment variables.

Monitoring with Prometheus

vLLM exposes Prometheus metrics at /metrics on the API server. Key metrics to track:

  • vllm:num_requests_running — currently active requests
  • vllm:num_requests_waiting — queued requests
  • vllm:gpu_cache_usage_perc — KV-cache memory utilization
  • vllm:avg_generation_throughput_toks_per_s — tokens generated per second
  • vllm:time_to_first_token_seconds — TTFT histogram

Point Prometheus to http://:8000/metrics, then visualize with Grafana. Set alerts for gpu_cache_usage_perc > 95% and num_requests_waiting > 100 to know when to scale up.

Hugging Face Token for Gated Models

Many popular models (Llama 3, Gemma, Mistral Large) require authentication. Log in to Hugging Face, generate a token at hf.co/settings/tokens, and pass it via:

$ export HF_TOKEN=hf_xxxxxxxxxxxxxxxxxxxxx
$ python -m vllm.entrypoints.openai.api_server \
    --model meta-llama/Meta-Llama-3.1-70B-Instruct \
    --trust-remote-code

In Docker, pass HF_TOKEN as an environment variable. vLLM will use it to authenticate with Hugging Face when downloading model weights.

When to choose what: vLLM is the best option when you need maximum throughput, multi-GPU support, or production-grade serving with metrics and hot-swappable LoRA adapters. TGI is the closest competitor to vLLM but currently lags in KV-cache efficiency and continuous batching flexibility.

Troubleshooting

CUDA Out of Memory

If vLLM crashes with a CUDA OOM error, reduce gpu_memory_utilization (try 0.80) or lower max_model_len. The model weights + KV-cache must fit entirely in GPU memory. Use nvidia-smi to check available memory before launching.

Model Not Found

Ensure the model name matches the Hugging Face repository exactly. For gated models, set the HF_TOKEN environment variable. If you're behind a proxy, configure HF_ENDPOINT or http_proxy/https_proxy.

Slow First Request

The first request after server start is slow because the model weights are being loaded and CUDA kernels are being compiled. Use --enforce-eager to skip CUDA graph optimization (faster startup, slightly slower throughput) or pre-compile with the Docker image.

Tokenization Errors

If you see mismatched tokenizer errors, pass --tokenizer-mode auto (default) and ensure the tokenizer is compatible with the model config. For trust_remote_code models, add --trust-remote-code.

NCCL Timeout on Multi-GPU

Set NCCL_TIMEOUT=600 and NCCL_SHM_DISABLE=0 to avoid initialization timeouts on systems with slow interconnects. For multi-node, ensure all nodes can communicate on the same network with NCCL_IB_DISABLE=1 as a fallback for non-Infiniband setups.
