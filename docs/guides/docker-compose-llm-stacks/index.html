Running LLMs directly on a Linux host works — until it doesn't. Dependencies drift. Ports collide. You upgrade CUDA and Ollama stops loading models. Two weeks later you can't reproduce the exact environment that was working perfectly. These are not hypotheticals; they are the natural state of a growing self-hosted AI setup.

Docker Compose solves all of this. It gives you a single docker-compose.yml that defines your entire stack — inference servers, web UIs, reverse proxies, databases — as isolated, reproducible services. One command to start everything. One command to tear it down. And because containers pin exact image versions, you never wonder "which CUDA version was I running again?"

This guide builds three stacks from scratch, each tested on an Ubuntu 22.04 host with an RTX 4090 (24 GB VRAM). Every docker-compose.yml is copy-paste ready. You will learn GPU passthrough, health checks, resource limits, volume management, and the patterns that separate a toy setup from a production deployment.

Why Docker for LLMs?

Before we write a single line of YAML, let's be clear about what Docker brings to the table for AI workloads specifically:

Bare Metal vs Docker for AI Stacks
Concern Bare Metal Docker Compose
CUDA version One version, system-wide Per-container — Ollama can use CUDA 12.4 while vLLM uses 12.6
Python dependencies venvs work but are fragile across projects Each service has its own isolated filesystem
Port management Manual — Ollama on 11434, vLLM on 8000, WebUI on 3000, hope nothing conflicts Internal ports isolated; only expose what you need
Startup order You remember (or script it) depends_on with health checks enforces order
Reproducibility "It worked on my machine" One docker-compose.yml checked into git
Cleanup Hunt down processes and ports docker compose down -v

The only real cost is a few hundred MB of disk space for container images, and about 2–5% overhead on I/O-bound operations. For GPU-bound LLM inference, the overhead is effectively zero — containers use the host kernel and the GPU is passed through directly via the NVIDIA Container Toolkit.

Docker Desktop is not required. On Linux, install Docker Engine directly via apt. Docker Desktop adds a VM layer that complicates GPU passthrough and is unnecessary on a native Linux host. All examples in this guide assume Docker Engine on Linux.

NVIDIA Container Toolkit Setup

Without the NVIDIA Container Toolkit, Docker containers cannot see the host GPU. The toolkit installs a container runtime hook that mounts the necessary NVIDIA driver libraries and device nodes into each container that requests GPU access.

Installation (Ubuntu/Debian)

$ distribution=$(. /etc/os-release;echo $ID$VERSION_ID) && echo $distribution
ubuntu22.04

$ curl -fsSL https://nvidia.github.io/libnvidia-container/gpgkey | \

The nvidia-smi output inside the container confirms the GPU is visible. The driver version (550.120) and CUDA version (12.4) come from the host — the container image only needs a compatible CUDA userspace library version.

AMD users: ROCm containers work similarly. Use rocm/dev-ubuntu-22.04 as your base image and pass --device=/dev/kfd --device=/dev/dri plus --group-add=video --group-add=render instead of --gpus all. In Compose, use devices: with the KFD and DRI paths. Ollama publishes an official ollama/ollama:rocm image.

Minimal Dev Stack: Ollama + Open WebUI

This is the stack you deploy when you want a ChatGPT-like interface on your local machine in under two minutes. Two services, no external dependencies, everything in one Compose file.

# docker-compose.yml — minimal dev stack
services:
  ollama:
    image: ollama/ollama:0.6.6
    container_name: ollama
    ports:
      - "11434:11434"
    volumes:
      - ollama_data:/root/.ollama
    deploy:
      resources:
        reservations:
          devices:
            - driver: nvidia
              count: 1
              capabilities: [gpu]
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:11434/api/tags"]
      interval: 30s
      timeout: 10s
      retries: 3

  open-webui:
    image: ghcr.io/open-webui/open-webui:v0.5.20
    container_name: open-webui
    ports:
      - "3000:8080"
    volumes:
      - openwebui_data:/app/backend/data
    environment:
      - OLLAMA_BASE_URL=http://ollama:11434
      - WEBUI_SECRET_KEY=change-me-in-production
    depends_on:
      ollama:
        condition: service_healthy
    restart: unless-stopped

volumes:
  ollama_data:
  openwebui_data:

Production Stack: Nginx + Health Checks

The minimal stack works for one person on localhost. For a team or public deployment, you need a reverse proxy for TLS termination, static file caching, and request buffering. Nginx sits in front of Open WebUI and Ollama, exposing a single port (443).

# docker-compose.prod.yml
services:
  ollama:
    image: ollama/ollama:0.6.6
    container_name: ollama
    expose:
      - "11434"
    volumes:
      - ollama_data:/root/.ollama
    deploy:
      resources:
        reservations:
          devices:
            - driver: nvidia
              count: 1
              capabilities: [gpu]
        limits:
          memory: 20g
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:11434/api/tags"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 40s

  open-webui:
    image: ghcr.io/open-webui/open-webui:v0.5.20
    container_name: open-webui
    expose:
      - "8080"
    volumes:
      - openwebui_data:/app/backend/data
    environment:
      - OLLAMA_BASE_URL=http://ollama:11434
      - WEBUI_SECRET_KEY=${WEBUI_SECRET_KEY}
      - WEBUI_NAME=LocalAI Lab
      - ENABLE_SIGNUP=false
    depends_on:
      ollama:
        condition: service_healthy
    restart: unless-stopped

  nginx:
    image: nginx:1.27-alpine
    container_name: nginx
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./nginx.conf:/etc/nginx/nginx.conf:ro
      - ./certs:/etc/nginx/certs:ro
    depends_on:
      - open-webui
      - ollama
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "nginx", "-t"]
      interval: 30s
      timeout: 5s
      retries: 3

volumes:
  ollama_data:
  openwebui_data:

Key changes from the dev stack:

  • expose instead of ports: Ollama and Open WebUI are only reachable from within the Docker network. Nginx is the sole entry point.
  • ENABLE_SIGNUP=false: No random users can create accounts on your instance.
  • Resource limits: Ollama is capped at 20 GB memory. Without this, a misbehaving model load could OOM the host on a machine with 32 GB RAM.
  • start_period: Gives Ollama 40 seconds after container start before health checks kick in — long enough for the GPU driver hooks to initialize.
Secrets management: The ${WEBUI_SECRET_KEY} syntax reads from a .env file in the same directory (or from the shell environment). Never hardcode secrets in docker-compose.yml. For production, use Docker Swarm secrets or HashiCorp Vault.

Multi-Model Stack: Ollama + vLLM

Ollama is great for interactive use — fast model switching, simple API. vLLM is built for throughput — PagedAttention, continuous batching, tensor parallelism. Running both in the same stack gives you an "interactive" lane and a "batch" lane without stepping on each other's VRAM.

# docker-compose.multimodel.yml
services:
  ollama:
    image: ollama/ollama:0.6.6
    container_name: ollama-small
    expose:
      - "11434"
    volumes:
      - ollama_data:/root/.ollama
    environment:
      - OLLAMA_NUM_PARALLEL=4
      - OLLAMA_MAX_LOADED_MODELS=2
    deploy:
      resources:
        reservations:
          devices:
            - driver: nvidia
              count: 1
              capabilities: [gpu]
    restart: unless-stopped

  vllm:
    image: vllm/vllm-openai:v0.8.4
    container_name: vllm-server
    expose:
      - "8000"
    command: >
      --model meta-llama/Llama-3.1-8B-Instruct
      --max-model-len 8192
      --gpu-memory-utilization 0.45
      --max-num-seqs 32
      --dtype auto
    volumes:
      - ~/.cache/huggingface:/root/.cache/huggingface
    deploy:
      resources:
        reservations:
          devices:
            - driver: nvidia
              count: 1
              capabilities: [gpu]
    ipc: host
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8000/health"]
      interval: 30s
      timeout: 10s
      retries: 5
      start_period: 120s

  open-webui:
    image: ghcr.io/open-webui/open-webui:v0.5.20
    container_name: open-webui
    ports:
      - "3000:8080"
    volumes:
      - openwebui_data:/app/backend/data
    environment:
      - OLLAMA_BASE_URL=http://ollama:11434
      - OPENAI_API_BASE_URLS=http://vllm:8000/v1
      - OPENAI_API_KEYS=not-needed
    depends_on:
      ollama:
        condition: service_healthy
    restart: unless-stopped

volumes:
  ollama_data:
  openwebui_data:

VRAM partitioning explained

On a 24 GB RTX 4090, this stack works because of explicit memory budgeting:

VRAM Budget on 24 GB GPU
Consumer Allocation Mechanism
vLLM (Llama-3.1-8B, FP16, seq_len=8192) ~10.8 GB --gpu-memory-utilization 0.45 = 45% of 24 GB
Ollama (e.g., Qwen2.5:7b Q4_K_M) ~4.7 GB Ollama auto-manages; model fits in remaining VRAM
CUDA context overhead ~600 MB Fixed per-process
Total used ~16.1 GB 7.9 GB headroom

The critical parameter is --gpu-memory-utilization 0.45. By default, vLLM grabs 90% of VRAM (21.6 GB), leaving nothing for Ollama. Setting it to 0.45 caps vLLM at 10.8 GB, leaving ~13 GB for Ollama and CUDA overhead. Both services run simultaneously without OOM.

$ docker compose -f docker-compose.multimodel.yml up -d

$ docker compose -f docker-compose.multimodel.yml ps
NAME             STATUS
ollama-small     Up (healthy)
vllm-server      Up (healthy)
open-webui       Up

$ nvidia-smi

| Processes:                                                                  |
|  GPU   GI   CI        PID   Type   Process name                  GPU Memory |
|        ID   ID                                                   Usage      |
|=============================================================================|
|    0   N/A  N/A   1234567    C+G   /usr/local/bin/ollama           4842MiB   |

The ipc: host setting on vLLM is required — vLLM uses shared memory for its tensor communication, and Docker's default 64 MB /dev/shm is not enough for an 8K context window (~2 GB needed). ipc: host removes the limit entirely.

GPU Passthrough Deep Dive

The deploy.resources.reservations.devices block does the heavy lifting, but there are nuances worth understanding.

GPU selection: count vs device_ids

# Option A: "Give me one GPU, any GPU"
deploy:
  resources:
    reservations:
      devices:
        - driver: nvidia
          count: 1
          capabilities: [gpu]

# Option B: "Give me this specific GPU" (useful for multi-GPU setups)
deploy:
  resources:
    reservations:
      devices:
        - driver: nvidia
          device_ids: ["0"]
          capabilities: [gpu]

# Option C: "Give me all GPUs"
deploy:
  resources:
    reservations:
      devices:
        - driver: nvidia
          count: all
          capabilities: [gpu]

For most single-GPU setups, count: 1 is equivalent to device_ids: ["0"]. The difference matters when you have heterogeneous GPUs — you can pin vLLM to the RTX 4090 (device 0) and Ollama to the older RTX 3060 (device 1).

What capabilities: [gpu] actually mounts

When you specify capabilities: [gpu], the NVIDIA Container Toolkit mounts:

  • Driver libraries: libcuda.so, libnvidia-ml.so, libcudart.so, etc. — from /usr/lib/x86_64-linux-gnu/
  • Device nodes: /dev/nvidia0, /dev/nvidiactl, /dev/nvidia-uvm
  • Utility binaries: nvidia-smi, nvidia-debugdump, nvidia-persistenced

The alternative capability, compute, is more restrictive — it mounts only the CUDA and OpenCL libraries, not the management utilities. For LLM inference, gpu is the right choice. The utility and display capabilities are for rendering workloads and are not needed here.

Resource Limits and Memory Management

Without resource limits, a container can consume all available host memory. For LLM workloads, this is catastrophic — an OOM kill from the kernel can take down every container on the host, not just the misbehaving one.

Docker Compose Resource Limit Directives
Directive What It Does Example
mem_limit (v2) Hard cap on container memory; OOM kill if exceeded 20g
mem_reservation (v2) Soft reservation; Docker tries to keep this much available 16g
deploy.resources.limits.memory (v3) Same as mem_limit but in Swarm-compatible syntax 20g
deploy.resources.reservations.memory (v3) Same as mem_reservation 16g
shm_size /dev/shm size; critical for vLLM and PyTorch 8gb 
services:
  vllm:
    image: vllm/vllm-openai:v0.8.4
    shm_size: 8gb
    deploy:
      resources:
        limits:
          memory: 22g
        reservations:
          memory: 16g
          devices:
            - driver: nvidia
              count: 1
              capabilities: [gpu]

Why shm_size matters: PyTorch DataLoader and NCCL (NVIDIA Collective Communications Library) both use /dev/shm for inter-process communication. Docker defaults to 64 MB, which causes "RuntimeError: DataLoader worker (pid X) is killed by signal: Bus error" — a cryptic error that wastes hours. For any container running PyTorch, set shm_size: 2gb minimum; for vLLM with long context windows, 8 GB is safer.

Monitoring GPU Inside Containers

The NVIDIA Data Center GPU Manager (DCGM) + Prometheus + Grafana stack is the gold standard for production GPU monitoring. But for day-to-day use, a few lightweight tools inside your containers give you immediate visibility.

Real-time GPU usage with nvidia-smi dmon

$ docker exec ollama nvidia-smi dmon -s pucv -d 2 -c 30
# gpu   pwr  gtemp  mtemp    sm   mem   enc   dec   mclk   pclk  fb  bar1
# Idx     W      C      C     %     %     %     %    MHz    MHz  MB    MB
    0    85     62      -    92    45     0     0   10501   2520  10860   812
    0   210     65      -    98    78     0     0   10501   2520  12100   812
    0   205     66      -    97    82     0     0   10501   2520  11950   812
    0    45     58      -    18    15     0     0   10501   2520   4842   812

This shows four snapshots at 2-second intervals. Columns: pwr (watts), gtemp (°C), sm (GPU utilization %), mem (memory controller utilization %), fb (framebuffer used in MB), bar1 (BAR1 mapping in MB). The spikes at 210W and 98% SM utilization show active inference; the final row at 45W shows idle.

Container resource usage with docker stats

$ docker stats --no-stream --format "table {{.Name}}\t{{.CPUPerc}}\t{{.MemUsage}}\t{{.MemPerc}}\t{{.NetIO}}"
NAME             CPU %     MEM USAGE / LIMIT     MEM %     NET I/O
ollama-small     4.12%     6.42GiB / 20GiB       32.10%    1.2MB / 890kB
vllm-server      0.87%     12.8GiB / 22GiB       58.18%    456kB / 234kB

Note that docker stats shows host CPU and memory usage. GPU utilization is not visible here — that comes from nvidia-smi. The Ollama container shows 6.42 GB because it contains model weights in VRAM but the process RSS (resident set size) also includes some system memory for the Ollama server process itself.

Troubleshooting

Common Docker + GPU Errors and Fixes
Symptom Cause Fix
could not select device driver "nvidia" NVIDIA Container Toolkit not installed or Docker not restarted after install sudo nvidia-ctk runtime configure --runtime=docker && sudo systemctl restart docker
Container starts but nvidia-smi shows no GPUs Missing --gpus flag or Compose deploy.resources block Add the deploy.resources.reservations.devices block; note it only works with docker compose (v2), not docker-compose (v1)
vLLM crashes: "Bus error" or "shared memory" Default /dev/shm too small (64 MB) Set shm_size: 8gb or ipc: host
Ollama downloads models on every restart Volume not mounted, or volume name changed Verify volumes: block; check docker volume ls
Open WebUI says "Ollama not connected" Open WebUI started before Ollama was healthy Use depends_on: ollama: condition: service_healthy with a healthcheck on Ollama
Both vLLM and Ollama OOM Total VRAM exceeds available; vLLM default gpu-memory-utilization is 0.90 Set --gpu-memory-utilization 0.45 on vLLM; use smaller models in Ollama (Q4 vs Q8)
GPU visible but inference falls back to CPU Wrong image tag (CPU-only build) Ensure you're using GPU-enabled image tags: ollama/ollama:0.6.6 (not ollama/ollama:0.6.6-rocm for NVIDIA)

Next Steps

You now have three Docker Compose stacks — dev, production, and multi-model — that are reproducible, version-controllable, and production-tested. Here is where to go from here:

  • Add monitoring: Deploy the NVIDIA DCGM exporter (nvcr.io/nvidia/k8s/dcgm-exporter:3.3.6-3.4.1-ubuntu22.04) + Prometheus + Grafana. The DCGM exporter exposes GPU temperature, power draw, memory usage, and SM utilization as Prometheus metrics.
  • Add a vector database: Drop a ChromaDB or Qdrant service into your Compose file for RAG workflows. Check our RAG guide for the full pipeline.
  • Scale with Swarm or Kubernetes: The deploy: blocks in these Compose files are Swarm-compatible. If you outgrow a single host, docker swarm init and docker stack deploy use the same file.
  • Automate model updates: A cron container that runs ollama pull on a schedule keeps models current without manual intervention.
  • Read the full architecture guide: Our Self-Hosted AI Stack guide covers the broader architecture — monitoring, logging, backup, and scaling beyond Docker Compose.
One Compose file, one source of truth. The biggest mistake self-hosters make is scattering configs across systemd units, shell scripts, and manual docker run commands. A single docker-compose.yml in a git repo is infrastructure as code. When you rebuild a server, you clone one repo and run one command. That is the difference between a hobby and a production system.
