1. What Is llama.cpp?

llama.cpp is a pure C/C++ inference engine for large language models (LLMs) that has become the de-facto standard for running LLMs on consumer hardware. Originally created by Georgi Gerganov as a proof-of-concept to run LLaMA models on a MacBook, it has grown into the most widely deployed self-hosted inference stack in the open-source ecosystem.

Why choose llama.cpp over alternatives like vLLM, TensorRT-LLM, or Hugging Face's Transformers?

  • Zero external dependencies — No Python, no PyTorch, no CUDA toolkit required for CPU inference. The entire engine compiles from source with nothing but a C++ compiler and Make or CMake. This makes it deployable on edge devices, Raspberry Pis, and stripped-down containers where Python stacks are impractical.
  • Full control — You own every inference parameter. There is no hidden preprocessing, no model-parsing abstraction layers that silently rewrite your config. Every flag is exposed on the CLI or server API, from temperature and top-p to mirostat, repetition penalty, and tail-free sampling.
  • GGUF format — llama.cpp introduced GGUF (GPT-Generated Unified Format), a single-file binary format that bundles weights, tokenizer, and hyperparameters. Unlike Hugging Face's multi-file safetensors + tokenizer.json + config.json sprawl, a GGUF file is self-contained and trivially version-managed.
  • Quantization parity — Quantization is a first-class feature, not an afterthought. llama.cpp supports more quantization formats than any other engine, from aggressive 2-bit quants suitable for CPU-only inference up to near-lossless 8-bit quants for GPU deployment.
  • Cross-platform — x86_64, ARM64 (Apple Silicon, Raspberry Pi), AMD GPUs via ROCm, NVIDIA GPUs via CUDA, Intel GPUs via SYCL, and almost any GPU via Vulkan compute. The same binary can be compiled for macOS, Linux, Windows, and even iOS/Android with minor adjustments.

According to the project's GitHub metrics, llama.cpp has accumulated over 100,000 stars and receives contributions from hundreds of developers. It powers everything from hobbyist chatbot UIs (Ollama, LM Studio, GPT4All) to production embedding pipelines for enterprise RAG systems.

2. Building from Source

Building llama.cpp from source gives you complete control over which backends are enabled and lets you apply local patches. The build system uses CMake with a ggml backend layer that abstracts hardware acceleration.

CPU-Only Build (Universal)

The simplest build compiles for CPU inference with no GPU acceleration. On modern x86_64 processors this enables AVX2, FMA, and BLAS optimisations automatically:

git clone https://github.com/ggerganov/llama.cpp
cd llama.cpp
cmake -B build -DCMAKE_BUILD_TYPE=Release
cmake --build build --config Release -j $(nproc)
# Binaries: ./build/bin/llama-cli, ./build/bin/llama-server, etc.

CUDA (NVIDIA GPUs)

For NVIDIA GPUs, enable the CUDA backend with offloading control. The GGML_CUDA_MMV_Y parameter controls batch size for matrix-vector operations — tuning this from 2 to 8 can yield 10–20% throughput improvements depending on your GPU architecture:

cmake -B build -DCMAKE_BUILD_TYPE=Release \
  -DGGML_CUDA=ON \
  -DGGML_CUDA_MMV_Y=4 \
  -DGGML_CUDA_FORCE_MMQ=ON
cmake --build build --config Release -j $(nproc)
# Use -ngl N to offload N layers to GPU at runtime

Metal (Apple Silicon)

On MacBooks and Mac Studios with M-series chips, Metal acceleration uses the unified memory architecture to eliminate CPU–GPU copy overhead. In practice this means you can offload 100% of layers without the VRAM ceiling that discrete GPUs impose:

cmake -B build -DCMAKE_BUILD_TYPE=Release \
  -DGGML_METAL=ON
cmake --build build --config Release -j $(sysctl -n hw.logicalcpu)

Vulkan and ROCm

Vulkan provides portable GPU compute across AMD, NVIDIA, and Intel GPUs. ROCm targets AMD GPUs natively. Both follow the same pattern:

# Vulkan
cmake -B build -DGGML_VULKAN=ON
# ROCm (AMD)
cmake -B build -DGGML_HIP=ON -DCMAKE_C_COMPILER=clang -DCMAKE_CXX_COMPILER=clang++

cmake --build build --config Release -j $(nproc)

You can combine multiple backends in a single build — for example, CUDA + Vulkan for heterogeneous GPU setups — though in practice the scheduler picks one backbone at runtime based on --main-gpu and --tensor-split flags.

3. GGUF Quantization Explained

GGUF quantization maps 16-bit (FP16) or 32-bit (FP32) floating-point weights into lower bit representations. llama.cpp implements two families of quantisation schemes: K-quants (legacy, mature) and IQ-quants (newer, higher quality at equivalent bitrates).

K-Quants

K-quants are block-wise quantization methods that handle outlier values by storing them at higher precision within each block. The naming convention is Q{b}{_variant}:

Format Bits/Weight Block Size Typical PPL Increase Use Case
Q2_K 2.56 256 +3.5–5.0% Minimum viable inference on 4 GB RAM
Q3_K_S / Q3_K_M / Q3_K_L 3.30–3.55 256 +2.0–3.5% Old hardware, sub-6 GB VRAM
Q4_K_S / Q4_K_M 4.25–4.50 256 +0.5–1.5% Sweet-spot for CPU inference
Q5_K_S / Q5_K_M 5.06–5.33 256 +0.2–0.5% Quality-conscious CPU deployment
Q6_K 6.56 256 +0.1–0.2% Near-lossless on CPU
Q8_0 8.00 32 < 0.1% GPU offload, embedding quality

The perplexity increases are measured against FP16 base on a standardised test set (WikiText-2, Llama 3 8B). Real-world degradation is typically lower than these benchmarks suggest because LLMs are highly robust to weight noise — a Q4_K_M model often produces text indistinguishable from FP16 in casual use.

IQ-Quants (Importance Quantization)

IQ-quants represent a step-change in quality-per-bit. Instead of uniform block-wise quantisation, IQ-quants use importance-weighted grouping: weights that contribute more to the model's output distribution are allocated extra bits. The result is significantly lower perplexity at aggressive quant levels:

Format Bits/Weight Effective Quality Notes
IQ1_S / IQ1_M 1.56–1.75 Approx Q3_K Experimental, use only for 7B+ on 4 GB
IQ2_XXS / IQ2_XS / IQ2_S / IQ2_M 2.06–2.70 Approx Q4_K_S Best quality-per-bit ratio today
IQ3_XXS 3.06 Approx Q5_K Excellent for low-resource servers
IQ4_NL / IQ4_XS 4.25–4.50 Approx Q6_K Near-lossless at 4-bit

Recommendation: For new deployments, always prefer IQ-quants over K-quants at the same bit budget. An IQ2_M (2.70 bpw) model achieves comparable quality to Q4_K_S (4.25 bpw) while using 36% less memory — this can mean the difference between running a 13B model on 8 GB of RAM vs needing 12 GB.

To quantize a model yourself:

# Quantize a GGUF FP16 model to IQ2_M
./build/bin/llama-quantize \
  --model /models/llama-3-8b-f16.gguf \
  --output /models/llama-3-8b-iq2m.gguf \
  --type IQ2_M
  --allow-requantize

# Check resulting file size
ls -lh /models/llama-3-8b-iq2m.gguf

4. CLI Inference Modes

llama.cpp ships three primary entry points: llama-cli for direct command-line inference, llama-server for HTTP API serving, and llama-perplexity for benchmarking. Understanding the CLI flags is essential for production tuning.

Interactive Mode

Interactive mode spawns a persistent session where both user input and model output stream in real time. This is ideal for testing prompts and sampling parameters before committing to a server configuration:

./build/bin/llama-cli \
  --model /models/llama-3-8b-iq2m.gguf \
  --interactive \
  --color \
  --ctx-size 4096 \
  --temp 0.7 \
  --top-p 0.9 \
  --repeat-penalty 1.1 \
  --mirostat 2 \
  --mirostat-lr 0.1 \
  --mirostat-ent 5.0

One-Shot Prompt

For batch or scripted use, pass the prompt directly and capture stdout. The --no-display-prompt flag omits the prompt echo, useful for piping output into other tools:

./build/bin/llama-cli \
  --model /models/llama-3-8b-iq2m.gguf \
  --prompt "Write a Python function to merge two sorted lists:" \
  --temp 0.2 \
  --ctx-size 2048 \
  --n-predict 512 \
  --no-display-prompt 2>/dev/null

Server Mode (HTTP API)

llama-server exposes an OpenAI-compatible API at /v1/completions, /v1/chat/completions, and /v1/embeddings. It supports multi-user concurrent requests with batching:

./build/bin/llama-server \
  --model /models/llama-3-8b-iq2m.gguf \
  --host 0.0.0.0 \
  --port 8080 \
  --ctx-size 8192 \
  --batch-size 512 \
  --ubatch-size 512 \
  --n-gpu-layers 35 \
  --parallel 4 \
  --cont-batching \
  --log-format json

The --cont-batching flag (continuous batching) allows the server to accept new requests while a generation is in progress, interleaving tokens across sequences. This is critical for production throughput — without it, concurrent requests queue and latency spikes under load.

5. Grammar-Based Structured Output

One of llama.cpp's most underrated features is its grammar engine, which constrains text generation to follow a formal grammar defined in GBNF (GGML BNF format). This eliminates JSON parsing errors, SQL injection risks in text-to-SQL pipelines, and hallucinated API response fields.

Grammar files are human-readable BNF-like rules. Here is a minimal JSON grammar (json.gbnf):

root ::= object
object ::= "{" pair ("," pair)* "}"
pair ::= string ":" value
string ::= "\"" [^"]* "\""
value ::= object | array | string | number | "true" | "false" | "null"
array ::= "[" value ("," value)* "]"
number ::= "-"? ("0" | [1-9] [0-9]*) ("." [0-9]+)? ([eE] [+-]? [0-9]+)?

Apply a grammar at inference time with --grammar-file:

./build/bin/llama-cli \
  --model /models/llama-3-8b-iq2m.gguf \
  --grammar-file /grammars/json.gbnf \
  --prompt "Generate a config object for a PostgreSQL connection" \
  --temp 0.3 \
  --n-predict 256

The model will never output a token that would violate the grammar — if the next token would break the JSON structure, the engine simply masks it out of the logits. This is fundamentally different from "parsing and retrying" approaches: it guarantees well-formed output in a single pass at zero extra latency.

For production JSON use, the project ships a complete JSON schema grammar converter at grammars/json_schema_to_grammar.py that translates any valid JSON Schema into a GBNF grammar, including nested objects, enums, and nullable fields:

python grammars/json_schema_to_grammar.py \
  --schema '{"type": "object", "properties": {"name": {"type": "string"}, "age": {"type": "integer", "minimum": 0}}}' \
  > /grammars/person.gbnf

6. Performance Tuning

llama.cpp exposes a wealth of runtime parameters that directly affect inference speed and memory usage. Understanding these knobs can double your throughput on the same hardware.

Context Size and Batch Size

--ctx-size controls the maximum number of tokens the model can attend to. Each token consumes ~2 bytes per layer of KV cache — for a 35-layer model at 8192 context, that's about 560 MB just for the cache. Setting --ctx-size too high wastes VRAM; setting it too low truncates useful context. Rule of thumb: set it to your prompt length + expected output length + 10% headroom.

--batch-size controls how many tokens are processed in a single forward pass during prompt ingestion. Larger batches (512–2048) maximise GPU utilisation during the prefill phase. For CPU inference, the optimal batch size is typically 256–512; beyond that, diminishing returns from memory bandwidth saturation set in.

--ubatch-size (micro-batch size) is the physical batch size sent to the backend. It should equal --batch-size in most configurations, unless you are using continuous batching on the server, in which case the micro-batch must be large enough to accommodate interleaved sequences.

Threading and Memory

Flag Default Effect Recommendation
--threads Physical core count Number of CPU threads for compute Set to nproc --all minus 1–2 for headroom
--threads-batch Same as --threads Threads used during batch prompt processing Often higher than --threads is beneficial
--no-mmap mmap enabled Disable memory-mapped model loading Use on systems with limited virtual address space (32-bit)
--mlock Off Lock model in physical RAM, prevent swapping Always enable in production servers
--numa Off Enable NUMA-aware memory allocation Essential on multi-socket servers (AMD EPYC, Intel Xeon)
--tensor-split N/A Split layers across multiple GPUs by ratio e.g. --tensor-split 2,1 for two GPUs with 2:1 capacity

Practical Tuning Workflow

Start with a baseline measurement using llama-perplexity or llama-cli --benchmark, then apply changes one at a time:

# Measure baseline tokens/second
./build/bin/llama-cli \
  --model /models/llama-3-8b-iq2m.gguf \
  --prompt "Hello" \
  --n-predict 256 \
  --benchmark 5 \
  --no-display-prompt 2>/dev/null

# Then tune: increase threads, enable NUMA, adjust batch
./build/bin/llama-cli \
  --model /models/llama-3-8b-iq2m.gguf \
  --prompt "Hello" \
  --n-predict 256 \
  --threads 8 \
  --threads-batch 16 \
  --batch-size 1024 \
  --numa \
  --benchmark 5 \
  --no-display-prompt 2>/dev/null

On an AMD EPYC 9654 (96 cores, dual-socket), enabling --numa alone yields a 15–25% throughput improvement by ensuring memory allocations stay on the local NUMA node. On a single-socket consumer CPU (Ryzen 7950X, Core i9-14900K), NUMA has no effect and can be omitted.

7. Embedding Server for RAG

llama.cpp can serve text embeddings via the same llama-server binary, making it a drop-in replacement for OpenAI's text-embedding-ada-002 endpoint. This is particularly valuable for RAG pipelines where you want to keep both generation and embedding inside your own infrastructure.

To enable the embedding endpoint, start llama-server with a model that has a <pooling> mode specified in its GGUF metadata (most instruct-tuned GGUF models include this):

./build/bin/llama-server \
  --model /models/bge-small-en-v1.5-q8_0.gguf \
  --embedding \
  --pooling mean \
  --port 8081 \
  --ctx-size 512 \
  --batch-size 512

The --pooling flag accepts three strategies:

  • mean — Average pooling over all non-padded tokens. Standard for sentence transformers.
  • cls — Use only the [CLS] token embedding. Useful for BERT-family models.
  • none — Return per-token embeddings (for sequence tagging tasks).

Query the endpoint with a standard OpenAI client:

curl http://localhost:8081/v1/embeddings \
  -H "Content-Type: application/json" \
  -d '{
    "input": "llama.cpp is a C++ inference engine",
    "model": "bge-small-en-v1.5"
  }' | jq '.data[0].embedding | length'

The response returns a 384-dimensional vector (for bge-small) or 768/1024 for larger embedding models. This plugs directly into vector databases like Chroma, Qdrant, or Milvus for semantic search.

Performance note: Embedding models are typically much smaller than generative models — a Q8_0 embedding model uses negligible VRAM and can run alongside a quantised generation model on the same GPU. Use a separate llama-server instance on a different port to avoid interference.

8. Converting HF Models to GGUF

Not every model on Hugging Face has a pre-built GGUF quant. When you need to convert a custom model or a recent release that hasn't been quantised yet, llama.cpp provides convert_hf_to_gguf.py (formerly convert.py).

The conversion pipeline has three stages:

  1. Download the HF model with git lfs or Hugging Face CLI.
  2. Convert to FP16 GGUF with convert_hf_to_gguf.py.
  3. Quantize the FP16 GGUF to your target format with llama-quantize.
# Stage 1: Download model from Hugging Face
git lfs install
git clone https://huggingface.co/mistralai/Mistral-7B-Instruct-v0.3
cd Mistral-7B-Instruct-v0.3
# Download the tokenizer files if not included
wget -O tokenizer.model https://huggingface.co/mistralai/Mistral-7B-Instruct-v0.3/raw/main/tokenizer.model

# Stage 2: Convert to FP16 GGUF
python llama.cpp/convert_hf_to_gguf.py \
  --outfile /models/mistral-7b-instruct-v0.3-f16.gguf \
  --outtype f16 \
  .   # Input directory containing the HF model

# Stage 3: Quantize to IQ2_M (1.7 GB instead of 14 GB)
./build/bin/llama-quantize \
  --model /models/mistral-7b-instruct-v0.3-f16.gguf \
  --output /models/mistral-7b-instruct-v0.3-iq2m.gguf \
  --type IQ2_M \
  --allow-requantize

# Verify
./build/bin/llama-cli \
  --model /models/mistral-7b-instruct-v0.3-iq2m.gguf \
  --prompt "Complete this: The capital of France is" \
  --temp 0 \
  --n-predict 10

Important Conversion Details

  • Tokenizer files: The converter handles SentencePiece (Llama, Mistral), BPE (GPT-2, Phi), and tiktoken (GPT-4, Llama 3). The tokenizer must be present in the HF repo as tokenizer.json or tokenizer.model — the converter does not download it from the Hub.
  • Config mapping: The converter reads config.json from the HF directory. Most architectures are supported out of the box (Llama, Mistral/Mixtral, Phi, Falcon, GPT-NeoX, DBRX, Qwen2, DeepSeek-V2). Unsupported architectures require a manual --model-class override.
  • Memory: FP16 conversion requires 2x the model's parameter memory. A 7B model needs ~14 GB of RAM. For large models (70B+), run the converter on a machine with 128 GB+ of RAM or use the --outtype q8_0 flag to quantize during conversion, reducing peak memory.
  • Split GGUF: For models too large to fit in a single file, GGUF supports split files with --split-max-size (e.g., --split-max-size 2G for 2 GB chunks), loadable with --model prefix-00001-of-000XX.gguf.

After conversion, always verify the model's output sanity with a known prompt before deploying. The --temp 0 (greedy decoding) flag eliminates randomness, making it easy to spot conversion corruption — a FP16 model and its quantised version should produce identical outputs for simple factual prompts.
